# Demo 技术架构

本文档用于说明本项目 Demo 的技术架构、模块边界、数据流和后续扩展方式。它面向团队内部协作，尤其用于后端开发、前后端联调和答辩时解释系统设计。

## 1. 项目定位

本项目 Demo 的核心目标不是做一个普通茶叶问答机器人，而是做一个“中国茶感知与文化表达的分层翻译系统”。

系统要解决的问题是：

```text
中国茶的成分、工艺、风味和文化经验，如何被结构化、解释清楚，并转译成国内消费者和海外受众都能理解的表达。
```

Demo 当前主路径为：

```text
3 款茶（铁观音 / 大红袍 / 金骏眉）× 图片物料 ×（国内链 + 跨文化链）两条同构链路
```

两条链路均完整跑通：国内链面向国内消费者，跨文化链面向欧美精品咖啡爱好者。两条链共享同一款茶的知识与风味坐标，跨文化表达由国内表达按规则横向翻译派生而来。其他市场、受众参照系、视频生成等路径先通过配置和 fallback 预留。

> **部署形态**：Docker 一体化（`docker-compose.yml`：backend FastAPI:8000 + frontend nginx:8080）。nginx serve `/frontend` 静态原型（`desktop-v2.html` / `mobile-v2.html` + `api.js`）并反代 `/api` `/docs` 到 backend，前端同源调无跨域。镜像构建时自动跑 `seed.py --reset` 灌表，密钥经 `./backend/.env`（gitignored）注入。详见 `deploy/nginx.conf` 与 README「Docker 复现」。
>
> **前端 v2**：`frontend/api.js` 封装 `BAMA_API`（`init` 拉 `/api/teas` 建茶名→`tea_id` 映射，`getTeaId` 精确匹配、找不到即抛错不兜底）；`desktop-v2.html` / `mobile-v2.html` 通过 `BAMA_API` 调全链路接口，枚举值经 `app/enum_map.py` 翻译。已取代 v1 mock 原型（硬编码知识库 + 假生成，已删除）。已知两缺口：①`api.js` `BASE` 写死 `localhost:8000`，nginx 同源部署需改相对路径 `""`；②v2 暂未读 `meta.fallback`，遇 fallback 显示空内容。
>
> **CI/CD**：GitHub Actions（`.github/workflows/deploy.yml`）在推 `main` 时 SSH 登服务器跑 `scripts/deploy-remote.sh`——`git reset --hard origin/main` + `docker compose up -d --build`。密钥经仓库 Secret（`SSH_HOST` / `SSH_PRIVATE_KEY`）注入，`backend/.env` 不进 Git、CI 不动它（仅在服务器上手动放一次）。详见 README「CI/CD」。

## 2. 总体架构

系统采用“四层功能 + 三类底座”的架构。

四层功能按递进逻辑组织：

```text
第 1 层：知识 / 证据层   （成分：茶品事实、工艺、成分、文化）
        ↓
第 2 层：风味结构化层     （感知：成分 → 风味坐标）
        ↓
第 3 层：表达生成层       （具象化：感知 → 可理解话术）
        ↓
第 4 层：营销物料层       （多模态物料：表达 → 海报 / 雷达 / image_prompt）
```

四层功能在第 3 层和第 4 层各自分出国内、跨文化两条同构链路，共享第 1、2 层的茶品事实。跨文化链由国内链按规则横向翻译派生而来。详见 3.3、3.4 节。

每一层都必须满足两个要求：

1. 能独立输出结果。
2. 输出结果能追溯到上一层依据。

因此系统不是一条写死的流水线，而是一组可组合模块。业务人员可以只查看知识卡片，也可以只生成跨文化表达，或者继续生成图片物料。

三类底座支撑四层功能：

```text
知识库：系统知道什么，包括茶品、产地、工艺、成分、风味术语、文化概念和业务调研洞察。
规则库：系统如何判断，包括什么时候可以类比、什么时候必须保留原概念、哪些表达不能写成事实。
工作流：系统按什么步骤完成任务，包括检索知识、筛选规则、构造 prompt、生成结果、校验输出和写入追溯链。
```

因此本项目不是单纯给 LLM 喂茶叶知识，而是把团队总结的方法拆成“数据、规则、工作流”，再由后端编排 LLM 完成表达转译。

## 3. 四层功能说明

### 3.1 第 1 层：知识 / 证据层

目标：

把茶品基础信息、产地、工艺、文化故事、公开文献代理数据、标准术语等内容结构化。

主要输入：

```text
tea_id
```

主要输出：

```text
茶品知识卡片
数据来源
工艺信息
文化故事
成分说明
```

对应接口：

```http
GET /api/teas/{tea_id}/knowledge
```

注意：

Demo 阶段如无八马内部检测数据，成分信息必须标注为“公开文献代理数据”或“典型范围”，不能伪装成单品实测值。

### 3.2 第 2 层：风味结构化层

目标：

把“好喝”“香”“有韵味”等模糊感知，转成可展示、可比较的风味坐标。

主要输入：

```text
tea_id
knowledge evidence
```

主要输出：

```text
风味维度
强度值
中文标签
英文标签
依据 ID
```

示例维度：

```text
花香
鲜爽度
回甘与尾韵
焙火感
醇厚度
```

对应接口：

```http
GET /api/teas/{tea_id}/flavor-profile
```

前端用途：

```text
风味雷达图
风味卡片
后续表达生成的依据展示
```

### 3.3 第 3 层：表达生成层

目标：

基于结构化风味坐标、业务洞察和匹配规则，生成面向不同受众的可理解表达。

表达生成层分为两条同构的链路，在层级上地位对等：

```text
国内中文表达
跨文化英文表达
```

两条链路都以第 1 层知识和第 2 层风味坐标为事实源头（按 `tea_id` 共享同一份茶品事实），区别在于表达语言和受众。跨文化表达由国内表达按规则横向翻译派生而来，规则库中会明确“国内的某个表达应如何信达雅地翻译成国外表达”。

需要强调：这种翻译是**同层横向派生**，不是纵向追溯链的一层。事实依据是成分与感知（两条链共享），翻译只是表达层的语言转换。`source_expression_id` 字段记录翻译源文，供前端可选展示，但不进入纵向追溯链。

国内表达解决：

```text
滋味难表达
门店话术难复用
```

跨文化表达解决：

```text
出海难翻译
中国茶概念难被海外受众理解
```

对应接口：

```http
POST /api/teas/{tea_id}/domestic-expression
POST /api/teas/{tea_id}/cross-cultural-expression
```

Demo 阶段重点：

```text
铁观音 × 英语 × 欧美市场 × 精品咖啡爱好者
```

跨文化表达原则：

1. 可以用咖啡参照系降低理解门槛。
2. 不把中国茶文化概念完全拍扁成咖啡术语。
3. 对“观音韵”“岩韵”等概念保留文化解释。
4. 能类比时说明类比依据；不能类比时不硬凑。

这些原则不应长期硬编码在 Python 或超长 prompt 中。Demo 阶段应把它们整理为结构化规则，存入 seed 文件并导入 SQLite。运行时按任务、市场、受众和茶品术语筛选少量相关规则，再注入 prompt。

### 3.4 第 4 层：营销物料层

目标：

把表达层结果转成可展示、可传播的图片物料结构。

国内链与跨文化链各自纵向深化到各自的物料层。国内物料和跨文化物料同等重要、以相同方式生成，区别仅在语言和文化。跨文化链在表达层之后、物料层之前，多一步跨文化类比适配（如用咖啡风味作参照、附加文化叙事），该适配作为跨文化表达对象内部的子结构返回，不单独成层、不进入纵向追溯链。

物料层优先由后端输出结构化物料数据（海报文案 / 雷达 / image_prompt），真实出图走独立接口。

> 注：真实生图（豆包 Seedream，火山方舟 Ark）已接入 `POST /api/image/generate`，与 `marketing-asset` 两步联调（物料层只产 `image_prompt`，生图拆为独立接口）。`image_prompt` 退化为纯画面物体 + 产地线索（不写镜头/构图/光照/色调/氛围）；生图时按 `scene`（`closeup` 特写 / `landscape` 产地广角 / `product` 商品罐图）注入镜头与构图、按 `style`（`fresh` 默认清新 / `business` 商务）注入光照与氛围——`scene` × `style` 正交，同一茶 prompt 可组合 6 种出图方向，无 seed 爆炸、不调 LLM、确定性，缓存键含两者防投毒。图源从 CogView-4 切换为 Seedream：CogView 中文文字渲染能力不稳，而海报初衷是把"知识点 + 产品文案"印在图上；Seedream 图内中文渲染准确，传 `tea_id + language` 时后端从 seed asset 表取 copy（`headline/subheadline/body`）直接渲染进图（full-bleed 无白边、带暗色渐变），不传则纯画面出图。`language` 进入缓存键（同 prompt 不同语言印不同文字，不进键会串成错语言图）。同时清除了企业画册美学信号词（`Professional commercial product photography` 等）。详见 docs/接口文档.md §6.2。

主要输出：

```text
海报标题
副标题
正文文案
风味雷达图数据
图片生成 prompt
追溯 ID
```

对应接口：

```http
POST /api/teas/{tea_id}/marketing-asset
```

可选后续能力：

```http
POST /api/image/generate        # 已实现（豆包 Seedream 真实生图 + 图内渲染中文知识文字，见 §3.4 说明 + 接口文档 §6.2）
POST /api/teas/{tea_id}/video-asset
```

`POST /api/image/generate` 已从"可选后续"升级为已实现（豆包 Seedream）；`video-asset` 仍为 Demo 阶段 fallback。

## 4. 技术栈

### 4.1 后端

```text
FastAPI
```

职责：

```text
提供 REST API
组织业务逻辑
查询数据（运行时查 SQLite，data_loader getter 经 seed.py --reset 灌表）
调用 LLM（已接入，未启用时走 seed 兜底）
处理 fallback
生成 Swagger 文档
```

### 4.2 数据库

```text
SQLite + SQLAlchemy（已接入）
```

职责：

```text
存储茶品、知识、风味坐标、术语、策略、Demo 路径和追溯关系
generated_outputs 表作 LLM 输出缓存（写路径）
```

运行时读路径查 `backend/data/tea.db`（由 `seed.py --reset` 从 YAML 灌表）；YAML 仍是数据源头，`.db` 不手动维护、被 gitignore。未灌表时读路径 best-effort 降级为空/None（不 500），启动打印警告。

### 4.3 种子数据

```text
YAML 或 JSON seed 文件 + seed.py
```

职责：

```text
YAML 是数据源头，可提交 Git、方便团队协作修改
seed.py --reset 可复现地从 YAML 灌进 SQLite（读真源）
generated_outputs 表随 LLM 调用写入（输出缓存）
```

推荐优先使用 YAML，因为茶文化故事、策略说明和表达规则包含较多长文本。

### 4.3.1 证据元数据

Demo 阶段数据量不大，所有关键资料采用轻量人工编码方式记录来源和可信度。Agent 可以辅助收集、整理和格式化，但 `source_type`、`source`、`confidence` 的最终判断由团队成员人工确认。

推荐每条关键资料至少包含：

```yaml
id: evidence_001
claim: "氨基酸与茶汤鲜爽感相关"
source_type: paper
source: "具体论文、标准、官网页面、访谈记录或社媒观察说明"
confidence: high
collected_by: "成员姓名"
notes: "限制、假设或使用场景"
```

`confidence` 建议人工按以下标准判断：

```text
high：官方标准、品牌官网、公开论文，或团队已明确决策。
medium：小样本访谈、社媒观察、行业文章，或多来源交叉印证但不严格。
low：单一非权威来源、个人经验、尚未验证的推测。
```

这套字段不是复杂的数据治理系统，只是为了保证 Demo 数据可追溯、可解释、可协作。

### 4.4 AI 生成

```text
LLM API（OpenAI 兼容 SDK）
```

已接入，默认指向 GLM（经 `backend/.env` 的 `LLM_BASE_URL` / `LLM_MODEL` 配置），可切换至：

```text
豆包
通义千问
DeepSeek
任意 OpenAI-compatible API
```

职责：

```text
生成中文表达文本（story_style / scientific_style / emotional_style）
生成跨文化表达文本（literal_explanation / beginner_analogy / cultural_narrative + analogy_rules）
生成营销文案（headline / subheadline / body / image_prompt）
```

不负责的（始终由 seed 提供）：ID、trace_id、source_expression_id / source_translation_id、雷达数值、证据字段。

原则：

LLM 不负责事实创造，也不独立决定项目方法。事实依据来自结构化知识（YAML seed 经 SQLite 查询）和风味坐标；判断边界来自结构化规则；后端工作流负责检索、筛选、组装和校验。未配置 key 或调用失败时透明退回 seed 预置表达。

### 4.5 缓存

已接入（写路径接库）：

```text
generated_outputs 表（SQLite）
```

后续可替换为：

```text
Redis
```

缓存对象：

```text
LLM 生成结果
跨文化表达结果
营销物料结果
```

已实现：`output_store` 把 LLM 生成结果按 `input_hash`（prompt + 输出 schema 类型）缓存进 `generated_outputs` 表；同输入命中即复用，跳过本次 LLM 调用。LLM 失败/未启用时不写缓存、退回 seed 兜底。data_loader 的 `@lru_cache` 仅缓存 seed registry 供 seed.py 灌表用。

### 4.6 前端

前端技术栈由前端组决定。

后端只需要保证：

```text
接口稳定
JSON 字段清晰
fallback 不让页面白屏
Swagger 文档可调试
```

**Docker 一体化部署**：`docker compose up -d --build` 拉起 `backend`（FastAPI:8000，构建时跑 `seed.py --reset` 灌表）+ `frontend`（nginx:8080，serve `/frontend` 静态并反代 `/api` `/docs` 到 backend）。云服务器只需暴露 frontend 的 8080，浏览器从同源调 `/api`，无跨域、无需 CORS。生图走豆包 Seedream 2K 常耗时 >90s，nginx 反代超时已放宽到 310s（对齐后端 `IMAGE_TIMEOUT=300`），避免网关先于后端 504。密钥经 `./backend/.env`（gitignored）注入容器，不进镜像。

**CI/CD**：`.github/workflows/deploy.yml` 在推 `main`（或 Actions 页手动触发）时，于 GitHub runner 用 `webfactory/ssh-agent` 注入部署密钥，SSH 登服务器执行 `scripts/deploy-remote.sh`：`git reset --hard origin/main` + `docker compose up -d --build`。并发控制 `concurrency: deploy-server`，新触发排队不中断进行中的部署。依赖仓库 Secret `SSH_HOST` / `SSH_PRIVATE_KEY`；`backend/.env` 不进 Git、CI 不动它（仅在服务器上手动放一次）。阿里云 ECS 须在安全组入方向放行 TCP 8080（OS 防火墙 `ufw inactive`、`iptables ACCEPT`，安全组是外网可达的唯一卡点）。

## 6. 数据策略

### 6.0 数据分层

本项目数据分为三类：

```text
科学信息：茶的组分、风味、产地、工艺、标准术语等相对确定的信息。
业务信息：不同文化受众理解差异、海外社媒观察、销售员访谈、茶叶爱好者访谈等调研信息。
规则信息：团队从调研和产品判断中总结出的可执行约束，例如类比边界、文化概念保留策略、营销事实边界和输出格式要求。
```

科学信息可以由 Agent 自动化收集公开资料并整理成 seed 文件，但关键结论需要团队人工确认来源和可信度。业务信息应以人类调研为主，Agent 负责辅助整理、打标签和转成 seed 格式。

规则信息与科学信息、业务信息一样进入 seed 文件和 SQLite，但用途不同：科学和业务信息回答“依据是什么”，规则信息回答“应该怎么判断和生成”。

### 6.1 SQLite 不是源数据

不要手工维护 `.db` 文件。

正确方式：

```text
seed YAML / JSON → seed.py → SQLite
```

Git 中提交：

```text
backend/data/seeds/*.yaml
backend/scripts/seed.py
```

Git 中忽略：

```text
backend/data/tea.db
```

这样可以保证任何人 clone 仓库后运行：

```bash
python scripts/seed.py --reset
```

都能得到同样的 Demo 数据库。`seed.py --reset` 是运行前置步骤：fresh clone 后必跑一次，否则启动会打印警告、读路径返回空/404。

### 6.2 Demo 路径配置

建议单独维护 `demo_routes.yaml`。

示例：

```yaml
routes:
  - id: szz_domestic_poster
    tea_id: BAMA_SZZ_TGY_NX
    market: domestic
    target_language: zh
    audience_reference: domestic_general
    asset_type: poster
    enabled: true
  - id: szz_western_coffee_poster
    tea_id: BAMA_SZZ_TGY_NX
    market: western
    target_language: en
    audience_reference: specialty_coffee_lovers
    asset_type: poster
    enabled: true
```

国内链与跨文化链均为主路径，各自走到物料层。两条链共享同一款茶的 `tea_id`，仅在 `market`、`target_language`、`audience_reference` 上不同。

后续扩展其他路径时，只需新增配置：

```yaml
  - id: dahongpao_western_wine_poster
    tea_id: dahongpao_001
    market: western
    target_language: en
    audience_reference: wine_lovers
    asset_type: poster
    enabled: true
```

### 6.3 Seed 文件中的证据字段

关键 seed 数据建议通过 `evidence_ids` 引用证据来源，而不是在每条业务数据里重复写长来源。

示例：

```yaml
flavor_profiles:
  - id: flavor_szz_tgy_nx
    tea_id: BAMA_SZZ_TGY_NX
    dimensions:
      - key: floral
        label_zh: 花香
        intensity: 8
        evidence_ids:
          - evidence_tea_flavor_wheel_2022
          - evidence_tieguanyin_public_description
```

对应证据：

```yaml
evidence_sources:
  - id: evidence_tea_flavor_wheel_2022
    claim: "茶叶感官风味轮可作为风味术语参考"
    source_type: public_standard
    source: "T/CTSS 58-2022 茶叶感官风味轮"
    confidence: high
    collected_by: "王传宇"
    notes: "用于 Demo 风味术语参考"
```

现阶段这些字段可以由团队成员手动填写。自动化脚本只负责校验字段是否存在、ID 是否能对应，不负责替团队判断可信度。

推荐的 `source_type` 取值：

```text
public_standard
paper
official_website
ecommerce
interview
social_media
team_assumption
industry_article
```

推荐的 `confidence` 取值：

```text
high
medium
low
```

### 6.4 规则数据

规则也应作为结构化数据维护。

推荐 seed 文件：

```text
backend/data/seeds/generation_rules.yaml
```

推荐字段：

```yaml
rules:
  - id: rule_cross_cultural_preserve_yun
    scope: cross_cultural_expression
    market: western
    audience_reference: specialty_coffee_lovers
    trigger_terms: ["观音韵", "韵"]
    priority: high
    instruction: "涉及观音韵时，不得直接替换成咖啡或葡萄酒术语。必须保留 Guanyin Yun，并提供入门类比和文化解释。"
    negative_example: "Guanyin Yun is the tannin of tea."
    positive_example: "Guanyin Yun can be introduced through a familiar lingering finish, but it is broader than a single coffee or wine note."
    enabled: true
  - id: rule_domestic_to_foreign_translation
    scope: cross_cultural_expression
    market: western
    audience_reference: any
    trigger_terms: []
    priority: high
    instruction: "跨文化表达由国内表达横向翻译派生而来。应将国内表达的中文 articulation 信达雅地转译为目标语言，保留原意与文化概念，不得脱离源文自由发挥。该翻译为同层横向派生，不作为纵向追溯链的一层。"
    negative_example: "忽略国内表达源文，凭空生成一段与源文无关的英文描述。"
    positive_example: "以国内表达中'兰花香、清爽、尾韵较长'为源文，转译为对应的英文表达。"
    enabled: true
```

字段说明：

```text
id：规则唯一 ID。
scope：适用任务，例如 cross_cultural_expression / domestic_expression / marketing_asset / any。
market：适用市场，例如 western / domestic / any。
audience_reference：适用受众参照系，例如 specialty_coffee_lovers / wine_lovers / any。
trigger_terms：触发术语，例如 观音韵、花香、回甘。为空时表示只要 scope/market/audience 匹配即可使用。
priority：high / medium / low。
instruction：真正注入 prompt 的规则说明。
negative_example：可选，错误示例。
positive_example：可选，推荐示例。
enabled：是否启用。
```

运行时不要注入全部规则。后端应根据当前请求筛选相关规则：

```text
1. scope 匹配当前任务或 any。
2. market 匹配当前市场或 any。
3. audience_reference 匹配当前受众或 any。
4. trigger_terms 与茶品术语、风味维度、文化概念有交集；若 trigger_terms 为空，则作为通用规则。
5. 按 priority 排序，取少量高相关规则。
```

示例：

```text
请求：铁观音 × 欧美市场 × 精品咖啡爱好者 × 生成介绍话术
茶品术语：铁观音、乌龙茶、兰花香、回甘、观音韵、清鲜

应筛选：
cross_cultural_expression 规则
western 规则
specialty_coffee_lovers 规则
触发 观音韵 / 花香 / 回甘 的规则
```

这样可以避免每次把全部规则传给 LLM，也能让规则随业务扩展逐步沉淀。

## 8. 数据流

### 8.1 知识卡片

```text
前端请求 tea_id
  ↓
后端查询 teas / tea_knowledge / evidence_sources
  ↓
返回知识卡片 JSON
```

### 8.2 风味坐标

```text
前端请求 tea_id
  ↓
后端查询 flavor_profiles / flavor_dimensions
  ↓
返回雷达图可用数据
```

### 8.3 国内表达

```text
前端提交 tea_id + 受众画像
  ↓
后端检查是否命中 demo_routes
  ↓
后端读取知识卡片 + 风味坐标 + 术语策略
  ↓
提取茶品术语、风味维度、文化概念
  ↓
从 generation_rules 筛选 domestic_expression 相关规则
  ↓
构造 Prompt：茶品事实 + 风味坐标 + 业务洞察 + 少量匹配规则 + 输出格式
  ↓
LLM 已启用 → 调用 LLM 生成文本；失败 / 未启用 → 退回 seed 预置 outputs
  ↓
写入 generated_outputs（按 input_hash 缓存，命中即复用，跳过 LLM 调用）
  ↓
trace_links 链路来自 seed（上一级指向风味坐标 source_profile_id）
  ↓
返回国内表达 JSON（ID / trace_id / source_profile_id 仍来自 seed）
```

### 8.4 跨文化表达

```text
前端提交 tea_id + market + target_language + audience_reference
  ↓
后端检查是否命中 demo_routes
  ↓
后端读取国内表达作为翻译源文
  ↓
后端读取知识卡片 + 风味坐标 + 术语策略（与国内链共享）
  ↓
提取茶品术语、风味维度、文化概念
  ↓
从 generation_rules 筛选相关规则（含“国内表达→国外表达”翻译规则、跨文化类比规则）
  ↓
构造 Prompt：国内表达源文 + 茶品事实 + 风味坐标 + 业务洞察 + 少量匹配规则 + 输出格式
  ↓
LLM 已启用 → 调用 LLM 生成文本；失败 / 未启用 → 退回 seed 预置 outputs
  ↓
写入 generated_outputs（记录 source_expression_id 为翻译源文，横向派生引用；按 input_hash 缓存）
  ↓
trace_links 链路来自 seed（纵向上一级指向风味坐标 source_profile_id；横向翻译关系不进 trace_links）
  ↓
返回跨文化表达 JSON（内含跨文化类比适配子结构，如 analogy_rules）
```

### 8.5 营销物料

国内链与跨文化链各自独立生成物料，方式相同，区别仅在语言和文化。

```text
前端提交 route_id + asset_type + language
  ↓
后端读取对应表达（国内表达或跨文化表达）+ 风味坐标
  ↓
从 generation_rules 筛选 marketing_asset 相关规则
  ↓
生成海报文案 copy + image_prompt（LLM）；雷达图数据始终来自 seed 事实，LLM 不碰
  ↓
写入 generated_outputs（按 input_hash 缓存）
  ↓
trace_links 链路来自 seed（纵向上一级指向对应表达 source_expression_id 或 source_translation_id）
  ↓
返回营销物料 JSON
```

## 9. LLM 调用边界

LLM 可以做：

```text
润色表达
生成多风格话术
生成跨文化解释
生成英文文案
生成图片 prompt（image_prompt 字段，非真实生图——真实生图由豆包 Seedream 走独立接口）
```

> 注：真实生图不调 LLM，由 `image_service` 直接调豆包 Seedream（火山方舟 Ark）文生图，并在发图前给 `image_prompt` 套确定性片段 + 图内中文文字（零幻觉）。故 `image_prompt` 由 LLM 生成、真实出图由 Seedream 完成，二者分离。图内中文知识文字由后端按 `tea_id + language` 从 seed asset 表取 copy（`headline/subheadline/body`）直接渲染进图（Seedream 中文渲染稳定）；不传 `tea_id + language` 则纯画面出图。

LLM 不应该做：

```text
编造成分数据
编造八马内部资料
编造标准条文
编造学术来源
生成 ID / trace_id / source_expression_id（由 seed 提供）
生成雷达数值（由 seed 事实提供）
```

Prompt 中必须明确：

```text
只能基于提供的知识、风味坐标和策略生成。
如果依据不足，使用保守表达。
不能声称代理数据是八马单品实测数据。
```

Prompt 不应长期包含全量规则。推荐做法是：

```text
system prompt：固定角色、总体安全边界、输出必须遵守 JSON。
task prompt：当前任务、茶品事实、风味坐标、业务洞察；跨文化表达时额外传入国内表达源文。
selected rules：从 generation_rules 动态筛选出的少量相关规则。
output schema：本接口要求的 JSON 字段（由 app/llm_schemas.py 的 Pydantic 模型定义，extra="forbid"）。
```

其中 `selected rules` 由后端规则筛选函数生成，而不是人工拼接一整份长规则文档。

降级策略（已实现）：LLM 未启用或调用失败时，文本字段透明退回 seed 预置 outputs，`meta.llm_generated=false` 并输出 `llm_fallback_reason`，但响应仍为 `success=true` 且 `fallback=false`（不误归为"功能未开放"）。降级原因取值共六类：尝试过 LLM 但失败 —— `network_error`（连接层失败：DNS / 握手 / 断流）/ `timeout`（请求超时）/ `parse_error`（LLM 输出非 JSON 或校验失败）/ `gateway_error`（网关返回 4xx/5xx：内容审查 / 额度 / 模型不存在 / 上下文超限等）；调 LLM 前源文缺失、未真正调用 —— `domestic_source_missing`（跨文化链取国内表达作翻译源文，但国内 seed 不存在）/ `expression_source_missing`（物料层取对应语言表达作文案依据，但该表达 seed 不存在）。后两类主路径不可达，仅作防御性兜底。完整的（情况 → fallback → llm_generated → llm_fallback_reason）映射见 `docs/接口文档.md §1.4`。

## 10. Fallback 设计

Demo 阶段有些功能会先由前端做好入口，但后端暂不开放真实能力，例如：

```text
视频生成
非英语目标语言扩展（如日语）
音频生成
非咖啡受众参照系
非 Demo 茶品完整链路
```

注意：真实生图（豆包 Seedream）已接入 `POST /api/image/generate`（见 §3.4 / 接口文档 §6.2），不再走 fallback；图源从 CogView-4 切换为 Seedream，图内渲染中文知识文字（传 `tea_id + language` 取 seed copy 印进图），出图质量已达标；市场与受众参照系枚举本身已开放（`GET /api/markets`、`GET /api/audience-references` 返回真实列表）；上述"未开放"指主路径之外的具体组合（如非欧美市场、非精品咖啡受众）在生成接口中会走 fallback。

后端应提供统一 fallback：

```json
{
  "success": true,
  "data": {
    "title": "功能暂未开放",
    "message": "该能力已在产品规划中，Demo 阶段暂不提供真实生成结果。",
    "available_route_id": "szz_western_coffee_poster"
  },
  "meta": {
    "demo_mode": true,
    "fallback": true,
    "fallback_reason": "feature_not_available"
  }
}
```

建议对 `/api/*` 的未知路由设置全局 fallback，避免前端出现白屏或默认 404。

## 11. API 分层

API 按优先级分三层：P0 必做（含 `domestic-expression` / `cross-cultural-expression` / `marketing-asset` / 真实生图 `image/generate` / `trace`）、P1 建议做（含 `natural-expression` NL 入口）、P2 占位即可（`video-asset` / `translate` / `audio/generate`，其中 `markets` / `audience-references` 已升级为真实列表）。`POST /api/image/generate` 已从 P2 升级为真实生图（豆包 Seedream，见 §3.4 / 接口文档 §6.2）。

完整的端点清单、字段定义、各端点的 P0/P1/P2 归属以 `docs/接口文档.md §10` 为准。

## 12. 可追溯机制

可追溯是本项目的重要技术亮点。

纵向追溯链按四层组织，国内链与跨文化链各自独立、相互对称。事实依据是成分与感知（两条链共享），翻译与类比是同层横向派生，不进入纵向追溯链。

国内物料追溯链：

```text
中文物料
  ↓
国内表达
  ↓
风味坐标
  ↓
知识依据 / 标准 / 公开资料
```

跨文化物料追溯链：

```text
英文物料
  ↓
跨文化表达
  ↓
风味坐标
  ↓
知识依据 / 标准 / 公开资料
```

跨文化表达由国内表达横向翻译派生而来，这一关系通过 `source_expression_id` 字段记录，供前端可选展示，但不作为纵向追溯链的一层。

前端可以通过：

```http
GET /api/trace/{output_id}
```

展示纵向依据链。

答辩时可以强调：

```text
系统不是让 AI 自由编写茶叶营销文案，而是用结构化知识约束 AI，把事实层、表达层和传播层串起来。
```

## 14. 风险与应对

### 14.1 数据真实性风险

风险：

没有八马内部检测数据。

应对：

明确标注公开文献代理数据，避免声称为单品实测。

### 14.2 LLM 输出不稳定

风险：

LLM 返回格式不稳定，影响前端解析。

应对：

使用 Pydantic schema 校验（`llm_schemas.py`，`extra="forbid"`）；校验失败 / 解析失败 / 网络 / 网关错误统一退回 seed 预置表达，并输出 `llm_fallback_reason` 供前端区分降级原因。

### 14.3 规则膨胀风险

风险：

规则越来越多，如果全部硬编码进 prompt，会造成 token 浪费、延迟增加和规则冲突。

应对：

规则结构化存储在 `generation_rules` 中；运行时按任务、市场、受众和触发术语筛选少量相关规则；必要时增加 priority 和 enabled 控制。

### 14.4 Demo 过度扩张

风险：

茶品、语言、市场和物料类型过多，导致十天内无法完成。

应对：

主路径只做一条，其他全部 fallback。

### 14.5 前后端字段不一致

风险：

前端写死字段后，后端改字段导致页面异常。

应对：

以 `docs/接口文档.md` 为准；字段变更必须同步更新文档。

### 14.6 CI/CD 密钥与 .env 泄漏

风险：

部署私钥或 `backend/.env`（含 `LLM_API_KEY` / `IMAGE_API_KEY`）误入 Git，密钥泄露。

应对：

部署私钥只存 GitHub 仓库 Secret（`SSH_PRIVATE_KEY`），公钥在服务器 `authorized_keys`；`.gitignore` / `.dockerignore` 双重挡 `backend/.env`，CI 脚本不跑 `git clean -fdx`（会误删被 gitignore 的 `.env`），`backend/.env` 仅在服务器上手动放一次。

### 14.7 云服务器安全组与端口冲突

风险：

阿里云安全组未放行 8080 导致外网超时；与服务器上既有 compose 项目（如个人博客占 80/443）端口或容器名冲突。

应对：

安全组入方向放行 TCP 8080 / 0.0.0.0（OS `ufw inactive`、`iptables ACCEPT`，安全组是唯一卡点，且服务器 `curl localhost:8080` 正常不代表外网通）；demo compose 仅映射 `8080:8080`，容器名 `chayu-bama-*`、项目名 `chayu-bama` 与博客 `my-blog-*` 互不冲突。

## 15. 总结

本项目 Demo 的技术架构核心是：

```text
结构化知识库约束事实
结构化规则库约束判断
风味坐标承接感知
工作流负责任务拆解
LLM 负责在规则约束下表达转译
物料层负责传播展示
追溯链证明每个输出有依据
fallback 保证未开放功能也能稳定交互
```

这套架构既适合报名阶段快速完成 Demo，也能自然扩展到更多茶品、市场、语言和传播物料。
