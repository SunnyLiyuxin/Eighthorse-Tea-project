## 一、后端接口总览

### 1.1 接口架构

后端基于 **FastAPI + SQLite**，采用四层纵向追溯链架构：

```
知识层 (Level 0) → 风味层 (Level 1) → 表达层 (Level 2) → 物料层 (Level 3)
```

两条同构链路：
- **国内链**：中文表达 + 中文物料
- **跨文化链**：英文表达 + 英文物料（从国内表达横向翻译派生）

### 1.2 核心 P0 接口（10个）

| 方法 | 路径 | 模块 | 说明 |
|------|------|------|------|
| GET | `/api/demo-routes` | 茶品 | 获取支持的Demo路径 |
| GET | `/api/teas` | 茶品 | 茶品列表 |
| GET | `/api/teas/{tea_id}/knowledge` | 茶品 | 知识卡片 |
| GET | `/api/teas/{tea_id}/flavor-profile` | 茶品 | 风味坐标卡 |
| GET | `/api/teas/{tea_id}/component-flavor` | 茶品 | 成分→口感映射 |
| POST | `/api/teas/{tea_id}/domestic-expression` | 表达 | 生成国内中文表达 |
| POST | `/api/teas/{tea_id}/cross-cultural-expression` | 表达 | 生成跨文化英文表达 |
| POST | `/api/teas/{tea_id}/marketing-asset` | 物料 | 生成营销物料（海报文案+雷达图+prompt） |
| POST | `/api/image/generate` | 图片 | 真实图片生成（豆包Seedream） |
| GET | `/api/trace/{output_id}` | 追溯 | 输出纵向追溯链 |

### 1.3 通用响应格式

```json
{
  "success": true,
  "data": { ... },
  "meta": {
    "demo_mode": true,
    "fallback": false,
    "llm_generated": false
  }
}
```

## 三、核心差异分析

### 3.1 前置选择参数差异汇总

#### 文案模式：后端支持 vs 前端UI

| 维度 | 后端（domestic-expression） | 前端UI | 差异 |
|------|---------------------------|--------|------|
| 受众年龄段 | `audience.age_group` (gen_z) | 无 | 后端有，前端无 |
| 茶知识水平 | `audience.knowledge_level` (beginner) | 入门/进阶/专业 | 概念一致，但枚举值需对齐 |
| 饮用场景 | `audience.scenario` (self_drinking) | 无 | 后端有，前端无 |
| 消费心理 | `audience.psychology` (curiosity) | 无 | 后端有，前端无 |
| 表达风格 | `style` (store_sales) | 语气+长度 | 概念不同，后端是预设风格，前端是语气+长度两个维度 |
| 时间节点 | 无 | 法定节日/节气/购物节/日常 | 前端有，后端完全没有 |
| 任务类型 | 无 | 成分→风味 / 模糊→形象 | 前端有，后端没有这个概念 |
| 风味参照系 | 无（国内版） | 咖啡/红酒/不参考 | 国内版后端无，海外版有 audience_reference |

#### 物料模式：后端支持 vs 前端UI

| 维度 | 后端（marketing-asset） | 前端UI | 差异 |
|------|------------------------|--------|------|
| 物料类型 | `asset_type` (poster) | 图片/视频 | 后端只有poster，视频是P2 fallback |
| 平台 | `platform` (wechat/tiktok) | 小红书/抖音/视频号 | 需做枚举映射 |
| 语言 | `language` (zh/en) | 中文/英文 | ✅ 对应 |
| 风格 | `style` (premium_but_approachable) | 年轻/商务/国风 | 枚举值完全不同，需对齐 |
| 内容主题 | 无 | 多种主题 | 前端有，后端无 |
| 目标消费者 | 无 | 入门/进阶/专业 | 前端有，后端无 |
| 时间节点 | 无 | 法定节日/节气/购物节 | 前端有，后端无 |

### 3.2 流程架构差异

#### 前端UI流程
```
首页 → 选择版本 → 选择文案/物料 → 前置选择（10+选项）
→ 对话窗口 → 逐步生成/调整 → 历史记录
```

#### 后端接口流程
```
选择茶品(tea_id) → 调用表达接口(expression) → 调用物料接口(marketing-asset)
→ 调用图片接口(image/generate) → 查询追溯(trace)
```

**核心差异：**
1. **前置选择维度数不同**：前端10+选项，后端只有3-5个参数
2. **粒度不同**：前端追求精细化调优（语气、长度、时间节点等），后端是固定产出
3. **对话式交互 vs API调用式**：前端是聊天式逐步调整，后端是一次性POST返回
4. **历史记录**：前端有，后端无
5. **自然语言入口**：后端有 `/api/natural-expression`，前端UI也有输入框，可以对接

## 四、对接方案建议

### 4.1 第一阶段：最小可用对接（MVP）

**目标**：让核心流程跑通，前端UI能拿到真实后端数据

#### 步骤1：茶品数据对接

- 调用 `GET /api/teas` 获取茶品列表
- 前端「茶叶品类」选项从接口动态获取
- 建立 UI 展示名 → `tea_id` 的映射（已落地：`api.js` `init()` 拉 `/api/teas` 建 name→id 映射，`getTeaId` 精确匹配、找不到即抛错不兜底）：
  - 赛珍珠浓香型安溪铁观音 → `BAMA_SZZ_TGY_NX`
  - 牛一武夷岩茶·大红袍 → `BAMA_NY_WRT_DHP`
  - 鼎红武夷红茶·金骏眉 → `BAMA_DH_BT_JJM`
  - ⚠️ 茶名须用后端 `/api/teas` 返回的**全名**，曾因前端用简称（「赛珍珠铁观音」等）与后端全名不匹配导致主路径全挂，已修复。改茶名须同步 v2 两 html 的 `TEA_CATEGORIES` 常量与物料调用默认值。

#### 步骤2：文案生成对接

- 国内版 → `POST /api/teas/{tea_id}/domestic-expression`
- 海外版 → `POST /api/teas/{tea_id}/cross-cultural-expression`
- 前端前置选择参数映射：
  - 「目标消费者」→ `audience.knowledge_level`（需对齐枚举值）
  - 「语气+长度」→ 合并映射到 `style`（需后端新增 style 枚举或前端做组合映射）
  - 「时间节点」→ **暂不支持，前端本地拼接到prompt中**（或作为后续需求提给后端）

#### 步骤3：物料生成对接

- 调用 `POST /api/teas/{tea_id}/marketing-asset`
- 参数映射：
  - 「平台」→ `platform`（小红书→wechat，抖音→tiktok，视频号→wechat）
  - 「风格」→ `style`（需后端扩展 style 枚举或前端做映射）
  - 「模态」→ `asset_type`（图片→poster，视频→fallback提示）

#### 步骤4：图片生成对接

- 先调 marketing-asset 拿到 `image_prompt`
- 再调 `POST /api/image/generate`，传入 prompt
- 前端展示 `image_url`

#### 步骤5：追溯功能对接

- 拿到 `trace_id` 后，调用 `GET /api/trace/{output_id}`
- 前端展示追溯链路

## 五、前端需要调整的事项

### 5.1 数据层改造

1. **新增 API 层**：封装所有后端接口调用（建议用 axios 或 fetch）
2. **状态管理调整**：Zustand store 中增加异步 action，从接口获取数据
3. **Mock数据保留**：开发环境下保留 mock 数据，接口不通时降级使用

### 5.2 前置选择组件调整

1. **选项数据从接口获取**：茶品列表、市场列表、受众参照系等
2. **参数映射层**：前端UI选项值 → 后端接口参数值的转换函数
3. **后端不支持的选项**：
   - 方案A：灰掉/隐藏（不推荐，影响UI完整性）
   - 方案B：保留展示，但提示"此功能暂未上线"（推荐，保留预期）
   - 方案C：前端本地模拟，把参数拼到 prompt 里传给后端（推荐，用户体验最好）

### 5.3 对话窗口调整

1. **消息结构对齐**：前端消息格式与后端响应格式做映射
2. **加载状态**：接口调用中显示 loading 状态
3. **错误处理**：后端返回 fallback 或 error 时的UI展示
4. **追溯功能入口**：在每条AI消息旁加「查看来源」按钮，调用 trace 接口

### 5.4 历史记录调整

1. **本地存储 → 云端存储**：从 localStorage 迁移到后端接口
2. **同步机制**：新建对话、切换对话、删除对话等操作对接后端
3. **离线降级**：接口不通时回退到本地存储
