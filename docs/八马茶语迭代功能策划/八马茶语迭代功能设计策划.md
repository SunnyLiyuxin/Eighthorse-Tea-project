# 第一章 系统定位与演进逻辑

## 1.1 当前定位：内部营销助手

目前demo即八马茶语 v1.0 ，是一套面向**八马茶业线上业务员**的智能营销助手，核心能力是帮助业务员快速生成符合品牌调性的茶业文案与物料。已完成的能力包括：

- 双版本（国内中文 / 海外英文）+ 双模式（文案 / 物料）工作台
- 前置选择（任务类型 / 品类 / 消费者 / 语气 / 长度 / 时间节点 / 风味体系）
- 图片生成（doubao-seedream-5-0-pro-260628）
- 本地历史记录 + 四层追溯链
- 电脑端 + 手机端双 UI

## 1.2 演进思考：为什么要从「内部助手」走向「全场景伙伴」

八马不是一家普通的茶企——是港股上市的**高端中国茶第一股**，品牌价值较高，全国 3800+ 门店，走进 32 国 87 城。这种体量决定了系统不能停留在"单人工具"阶段，必须沿三条主线演进：

| 演进主线 | 驱动力 | 落地版本 |
|---------|--------|---------|
| **协作化** | 单人工具无法沉淀组织资产，业务员流动会带走经验 | v2.0 |
| **生态化** | 八马已有飞书 AI 生态（数字旺财 / AI 图像核验 / AI 店群助手），系统不能是孤岛 | v2.0 |
| **出海化** | 港股上市后国际化是核心战略，海外业务员需要英文物料生产能力 | v2.0–v3.0 |
| **年轻化** | 主品牌高端定位不能激进创新，需子品牌试点 IP / 国潮 | v2.0–v3.0 |
| **培训化** | 海外扩张最大瓶颈是"人"，需把培训能力产品化 | v4.0 |
| **C 端化** | 高端茶决策门槛高，需把茶艺师能力规模化输出 | v4.0+ |

## 3.2 已知局限：产品库覆盖范围

> **本节是 v1.0 → v2.0 迭代的首要驱动**：当前 demo 的产品库仅适配三款最经典的茶，需在 v2.0 阶段优先补齐全 SKU。

### 3.2.1 当前覆盖范围

| 维度 | 现状 |
|------|------|
| 适配茶品数 | **3 款**（仅最经典的代表性茶品） |
| 数据库覆盖 | 仅 3 款茶的元数据、风味维度、表达策略 |
| 知识库覆盖 | 仅 3 款茶的产区、工艺、品鉴、品牌叙事 |
| 前端适配 | 前置选择器、风味雷达图、追溯链仅针对 3 款茶调优 |

### 3.2.2 为什么先做 3 款（demo 阶段合理性）

| 维度 | 说明 |
|------|------|
| 验证可行性 | 用最小产品集验证"AI 能否生成符合品牌调性的茶文案"这一核心假设 |
| 控制工程量 | 3 款茶足以跑通全链路（前置选择 → 知识检索 → 风味匹配 → 文案生成 → 物料生成 → 追溯），不需要一开始就投入全 SKU 数据建设 |
| 聚焦体验 | demo 阶段重点是交互体验与品牌质感，而非数据完整性 |

### 3.2.3 为什么 v2.0 必须补齐（生产可用门槛）

| 前提条件 | 说明 |
|---------|------|
| 业务员真实使用场景 | 业务员每天面对的是八马全 SKU（数百款），3 款茶无法支撑日常营销工作 |
| 共享云端素材库的前提 | 素材库的价值在于覆盖全品类的爆款物料，3 款茶无法形成规模效应 |
| 飞书生态联动的前提 | 与数字旺财、CRM、ERP 互通需要全 SKU 数据对齐 |
| 出海与培训的前提 | 海外员工培训 Agent 与 C 端导购 Agent 都需要全 SKU 知识库作为底座 |

# 第四章 v2.0 详细功能设计

让单人工具升级为团队工具，让系统融入八马已有生态，让出海走稳第一步，让年轻化有抓手。

## 4.1 产品库扩充（v1.0 → v2.0 首要任务）

> **本节是 v2.0 的第一项任务，也是 v1.0 → v2.0 迭代的桥梁**：把 demo 阶段的 3 款经典茶扩充到八马全 SKU，让系统从"可演示"升级为"可生产"。

### 4.1.1 扩充范围

| 维度 | v1.0 现状 | v2.0 目标 |
|------|-----------|----------|
| 适配茶品数 | 3 款经典茶 | **全 SKU**（八马主品牌 + 万山红子品牌） |
| 茶品类覆盖 | 部分 | 六大茶类全覆盖（绿/白/黄/青/红/黑）+ 再加工茶 |
| 价格带覆盖 | 部分 | 全价格带（口粮茶 / 礼盒茶 / 高端礼赠 / 收藏级） |
| 包装规格覆盖 | 部分 | 散茶 / 袋泡 / 礼盒 / 限量款全覆盖 |

### 4.1.2 三库同步扩充（数据库 + 知识库 + 前端）

| 库类型 | 扩充内容 | 工程量 |
|--------|---------|--------|
| **数据库** | 全 SKU 元数据（茶名 / 品类 / 产区 / 树龄 / 工艺 / 价格 / 包装 / SKU 编码） | 大 |
| **风味维度库** | 全 SKU 风味雷达图（香气 / 滋味 / 回甘 / 韵味 / 层次 / 体感） | 大 |
| **知识库** | 全 SKU 产区风土、加工工艺、品鉴要点、冲泡参数、储存养护、品牌叙事 | 大 |
| **表达策略库** | 全 SKU 适配的表达策略（按品类 / 价格带 / 场景） | 中 |
| **前端前置选择器** | 品类下拉、茶品下拉、风味雷达图渲染全面适配全 SKU | 中 |
| **前端追溯链** | 四层追溯（知识→风味→表达→物料）适配全 SKU | 中 |
| **图片生成提示词** | 全 SKU 的产品图 / 场景图 / 礼盒图提示词模板 | 中 |

### 4.1.3 扩充路径（分批推进，避免一次性投入过大）

| 批次 | 范围 | 时间 | 优先级 |
|------|------|------|--------|
| 第 1 批 | 八马主品牌核心 20 款（铁观音 / 大红袍 / 红茶 / 白茶 / 普洱主力 SKU） | v2.0 第 1 个月 | P0 |
| 第 2 批 | 八马主品牌全 SKU（含礼盒 / 限量款 / 收藏级） | v2.0 第 2 个月 | P0 |
| 第 3 批 | 万山红子品牌全 SKU（含侠客 IP 四款） | v2.0 第 3 个月 | P0 |
| 第 4 批 | 海外版英文物料适配的全 SKU | v2.0 第 3 个月（与第 3 批并行） | P1 |

### 4.1.4 数据建设保障

| 保障项 | 说明 |
|--------|------|
| 数据来源 | 八马既有产品数据库（CRM / ERP）+ 茶艺师人工补录 + 品牌部审核 |
| 数据质量 | 每款茶的风味维度需 ≥ 2 位资深茶艺师独立评定后取均值 |
| 数据治理 | 建立 SKU 数据 owner 制度，每个品类指定 owner 负责数据准确性 |
| 数据更新 | 新品上市 T+7 天内完成全库扩充；产品下架 T+1 天内标记下架 |

### 4.1.5 验收标准

| 验收项 | 标准 |
|--------|------|
| 前置选择器 | 全 SKU 可选、可生成文案、可生成物料 |
| 风味雷达图 | 全 SKU 渲染正确，无空值 |
| 四层追溯链 | 全 SKU 完整可查 |
| 业务员抽查 | 抽查 20 款茶（覆盖各品类各价格带），生成内容通过茶艺师审核 |

| 维度 | 语音识别方案 | 评论抓取方案 |

## 6.1 战略背景

### 6.1.1 海外扩张的人才瓶颈

八马招股书披露，国际化是上市后核心战略。但海外扩张最大瓶颈不是供应链、不是渠道，而是**人**：

| 痛点 | 说明 |
|------|------|

| 这款茶适合我吗？ | 不知道自己的口味偏好与茶的匹配度 |
| 它的风味特点是什么？ | 看不懂专业术语（如"岩骨花香""兰花韵"） |
| 它为什么值这个价？ | 不理解非遗技艺、年份稀缺等价值支撑 |
| 在什么场景下喝最合适？ | 不知道商务、送礼、自饮各该选什么 |
| 怎么冲泡才好喝？ | 不会控制水温、投茶量、冲泡时间 |

### 7.1.2 传统门店服务的局限

| 局限项 | 说明 |
|--------|------|
| 茶艺师数量有限 | 服务半径有限 |
| 高峰期客户等待时间长 | 体验差 |
| 茶艺师水平参差不齐 | 服务体验不一致 |
| 客户离店后无人持续服务 | 复购率低 |
| 线上电商完全没有茶艺师服务 | 线上体验断层 |

### 7.1.3 Agent 作为 C 端导购的核心价值

| 价值点 | 说明 |
|--------|------|
| 7×24 小时在线 | 把茶艺师专业能力规模化输出 |
| 专属茶艺师 | 让每一位消费者都有"专属茶艺师" |
| 全触点覆盖 | 线上电商 + 线下门店 + 私域社群 |
| 个性化推荐 | 持续学习消费者偏好 |
| 数据沉淀 | 反哺产品研发与营销 |

## 7.2 五大导购场景

| 场景 | 触发 | Agent 行为 |
|------|------|-----------|
| **场景一：选茶推荐** | 客户进入门店/电商 | 通过 3-5 个问题，推荐最适合的茶 |
| **场景二：风味讲解** | 客户查看某款茶详情 | 用通俗语言讲解风味特点 + 冲泡建议 |
| **场景三：价值传递** | 客户质疑价格 | 讲解非遗技艺、年份稀缺、品牌背书 |
| **场景四：场景匹配** | 客户说"我要送礼" | 推荐礼品茶 + 包装方案 + 附送卡片文案 |
| **场景五：售后陪伴** | 客户购买后 | 推送冲泡提醒、品鉴笔记、复购建议 |

# 第八章 三 Agent 协同矩阵 + 数据飞轮

## 8.1 三 Agent 协同矩阵

| 维度 | 营销助手 (v1-v3) | 培训 Agent (v4) | 导购 Agent (v4+) |
|------|-----------------|----------------|-----------------|
| **服务对象** | 业务员 | 新员工 | 消费者 |
| **核心场景** | 内容生产 | 知识培训 + 实时辅助 | 选茶推荐 + 售后陪伴 |
| **表达风格** | 专业 + 可配 | 教学化 + 循序渐进 | 通俗 + 亲切 |
| **交互深度** | 单次任务 | 长期学习路径 | 长期用户关系 |
| **数据归属** | 组织资产 | 员工能力档案 | 用户画像（合规） |
| **触达方式** | 工作台 | 培训 App + 飞书 | 全触点 |
| **底层复用** | 茶业知识库 + 风味库 + 表达库 | 同左 | 同左 |

## 8.2 数据飞轮

### 8.2.1 飞轮闭环流程

| 步骤 | 飞轮节点 | 数据流向 |
|------|---------|---------|
| 1 | 营销助手生成内容 | 内容成为培训 Agent 的教学素材 |
| 2 | 培训 Agent 收集员工常见问题 | 反哺营销助手优化表达 |
| 3 | 培训合格员工到一线服务 | 把能力规模化输出给消费者 |
| 4 | 导购 Agent 收集消费者真实反馈 | 反哺营销助手 + 培训 Agent |
| 5 | 产品研发根据消费者偏好数据 | 发现新品类机会 |
| 6 | 新产品上市 | 触发营销助手生成新内容，飞轮再次启动 |

### 8.2.2 飞轮数据流向示意

| 数据来源 | 数据内容 | 反哺对象 |
|---------|---------|---------|
| 营销助手 | 生成的文案 / 物料 | 培训 Agent（教学素材） |
| 培训 Agent | 员工常见问题 / 考核盲区 | 营销助手（优化表达） |
| 培训 Agent | 员工能力档案 | 一线服务（规模化输出） |
| 导购 Agent | 消费者真实反馈 / 提问 | 营销助手 + 培训 Agent |
| 产品研发 | 消费者偏好数据 | 新品研发 |
| 新产品 | 新 SKU 上市 | 营销助手（生成新内容） |

## 8.3 飞轮价值

| 飞轮价值点 | 说明 |
|-----------|------|
| 内容自动复用 | 营销助手生成的内容，自动成为培训 Agent 的教学素材 |
| 表达持续优化 | 培训 Agent 收集的员工常见问题，反哺营销助手优化表达 |
| 能力规模化 | 培训合格员工到一线服务，把能力规模化输出 |
| 消费者反馈反哺 | 导购 Agent 收集的消费者真实反馈，反哺营销助手 + 培训 Agent |
| 研发驱动 | 产品研发根据消费者偏好数据，发现新品类机会 |
| 飞轮自循环 | 新产品触发营销助手生成新内容，飞轮再次启动 |

**核心承诺**：三个 Agent **不是三个系统，是一个生态**——共用底层能力，数据互相反哺，越用越聪明。

# 第九章 关键技术决策说明

> **本章集中说明每个关键选择背后的思考**

## 9.1 直播方案：为什么选语音识别（不选评论抓取）

| 维度 | 语音识别方案（选定） | 评论抓取方案（远期） |
|------|-------------|-------------|
| 技术可行性 | ASR（豆包/讯飞）成熟，5 秒内出答案 | 平台反爬严，弹幕 API 不开放给第三方 |
| 主播可控性 | 主播主动筛选要回答的问题 | 被动接收评论，质量参差 |
| 底层复用 | 复用 natural-expression 接口 | 需新建抓取 + 清洗管道 |
| 风险 | 低 | 高（违反平台规则风险） |
| 上线版本 | v3.0 | v4+ 远期目标（需平台官方授权） |

## 9.2 侠客 IP：为什么先在万山红试点（不上主品牌）

| 维度 | 万山红试点（选定） | 主品牌直接上 |
|------|-------------|-------------|
| 品牌资产风险 | 隔离主品牌高端资产稀释风险 | 直接稀释主品牌高端定位 |
| 试错成本 | 万山红价格带中端，试错成本可控 | 主品牌试错成本高 |
| 人群契合度 | 万山红目标人群 18-30 岁年轻女性，对 IP/国潮天然亲近 | 主品牌 30-55 岁商务人群，IP 接受度未知 |
| 失败止损 | 6 个月评估节点可快速下线 | 失败影响主品牌声誉 |
| 成功路径 | 验证有效后逐级上提主品牌 | 无上提路径，直接 All-in |

## 9.3 出海战略：为什么先做英文通用版（不绑定单一国家）

| 维度 | 英文通用版（选定） | 单一国家深度定制 |
|------|-------------|-------------|
| 通用性 | 英文覆盖全球商务沟通、海外华人圈、国际社媒主流语言 | 单一国家覆盖面窄 |
| 内容生产闭环 | 不需要为每个小语种建独立语料库，工程量可控 | 需为每个国家建独立语料库与模型 |
| 品牌叙事统一 | 一套英文叙事打天下，避免品牌形象割裂 | 各国叙事不一致，品牌管理复杂 |
| 与八马既有履历匹配 | 八马已走进 32 国 87 城、款待 192 国宾客，英文物料足以承载 | 单一国家深度定制与"32 国"履历不匹配 |
| 扩展性 | v3.0 可视业务进展扩展日/韩/法语 | 需为每个新市场重新定制 |

## 9.4 Agent 演进：为什么分三段（不跳跃式扩张）

| 维度 | 三段式演进（选定） | 跳跃式扩张 |
|------|-------------|-------------|
| 风险控制 | 每一步都稳，每一步都验证 | 一次性铺开，风险集中 |
| 底层复用 | 三个 Agent 共用同一套底层能力 | 各 Agent 独立建设，重复造轮子 |
| 数据积累 | 内部助手先行积累数据，培训次之，导购最后 | 数据基础不足，C 端体验差 |
| 组织准备 | 内部团队逐步熟悉 Agent 运营 | 组织能力跟不上 |
| 商业逻辑 | B 端内 → B 端外 → C 端，符合企业服务客户路径 | 跨度太大，资源分散 |
