# Demo 接口文档

本文档用于前后端协作，描述 Demo 阶段后端 API 的功能边界、请求参数、响应结构和 fallback 规则。

Demo 当前主路径为：

```text
3 款茶（铁观音 / 大红袍 / 金骏眉）× 图片物料 ×（国内链 + 跨文化链）两条同构链路
```

两条链路均完整跑通：国内链面向国内消费者，跨文化链面向欧美精品咖啡爱好者。两条链共享同一款茶的知识与风味坐标，跨文化表达由国内表达按规则横向翻译派生而来，这一关系通过 `source_expression_id` 字段记录，不进入纵向追溯链。

后端会优先保证这两条链路完整跑通。其他市场、受众参照系、视频生成等能力可以先由前端做好入口，后端通过 fallback 返回"暂未开放"的标准响应，避免页面空白或报错。

> **部署形态**：Docker 一体化部署后，前端经 nginx 网关同源调 `/api`（无跨域）。`Base URL` 即 nginx 入口（默认 `http://localhost:8080` 或云服务器 `http://<IP>:8080`）；直连后端则 `http://localhost:8000`。下方示例统一以 `/api` 相对路径表示。

## 1. 通用约定

### 1.1 Base URL

本地开发默认（直连后端）：

```text
http://localhost:8000
```

Docker 一体化部署（推荐，经 nginx 反代，前端同源）：

```text
http://localhost:8080   # 或云服务器 http://<IP>:8080
```

所有业务接口统一以 `/api` 开头。

### 1.2 数据格式

请求和响应均使用 JSON：

```http
Content-Type: application/json
```

### 1.3 ID 约定

Demo 阶段建议使用可读字符串 ID，而不是纯数字 ID，方便前后端调试和追溯。

示例：

```text
BAMA_SZZ_TGY_NX
flavor_szz_tgy_nx
expr_cn_szz_tgy_nx
expr_en_szz_tgy_nx_coffee
asset_szz_poster_zh
asset_szz_poster_en
```

### 1.4 通用响应字段

正常响应建议包含：

```json
{
  "success": true,
  "data": {},
  "meta": {
    "demo_mode": true,
    "fallback": false,
    "llm_generated": true,
    "used_rule_ids": ["rule_domestic_store_sales_style"]
  }
}
```

生成类接口（5.1 / 5.2 / 6.1）的 `meta` 额外字段：

- `llm_generated`：本次输出是否由 LLM 真实生成。`false` 表示退回了 seed 预置表达（未启用或调用失败时的兜底）。
- `llm_fallback_reason`：仅当 **LLM 已启用（`llm_enabled=true`）但本次未产出 LLM 内容**（`llm_generated=false`）时出现。未启用 LLM 时 `llm_generated=false` 但**不输出**本字段（行为与未接 LLM 时一致）。取值分两类：**(1) 尝试过 LLM 但失败** —— `network_error`（连接层失败：DNS / 握手 / 断流）/ `timeout`（请求超时）/ `parse_error`（LLM 输出非 JSON 或校验失败）/ `gateway_error`（网关返回 4xx/5xx：内容审查 / 额度 / 模型不存在 / 上下文超限等）；**(2) 调 LLM 前源文缺失，未真正调用** —— `domestic_source_missing`（跨文化链取国内表达作翻译源文，但国内 seed 不存在）/ `expression_source_missing`（物料层取对应语言表达作文案依据，但该表达 seed 不存在）。两类均主路径不可达、Demo 阶段不会触发，仅作防御性兜底。注意：LLM→seed 降级**不**置 `fallback=true`，仍属成功响应，前端可直接使用 seed 文本展示。
- `used_rule_ids`：本次注入 LLM prompt 的结构化规则 id 列表，用于"规则约束生成"可追溯。

NL 入口（§5.3）的 `meta` 额外含 `nl` 子对象：`directive`（回显用户原始输入）、`intent_llm_generated`（意图解析是否走了 LLM，含缓存命中）、`chain`（路由到的链路 `domestic` / `cross_cultural`）。其余 `llm_generated` / `used_rule_ids` / `llm_fallback_reason` 语义与上面一致，描述的是话术生成阶段。

fallback 响应建议包含：

```json
{
  "success": true,
  "data": {},
  "meta": {
    "demo_mode": true,
    "fallback": true,
    "fallback_reason": "该功能 Demo 阶段暂未开放"
  }
}
```

错误响应建议包含：

```json
{
  "success": false,
  "error": {
    "code": "TEA_NOT_FOUND",
    "message": "未找到对应茶品"
  }
}
```


用途：

```json
        "label_en": "Freshness",

        "description": "入口清爽，回味较明亮",
        "evidence_ids": ["evidence_public_oolong_components"]
      },
      {
        "key": "finish",
        "label_zh": "回甘与尾韵",
        "label_en": "Lingering Finish",
        "intensity": 8,
        "description": "饮后口腔留香，尾韵较长",
```

- 雷达图可优先使用 `label_zh` 或 `label_en`。
- 如果需要展示依据，可以根据 `evidence_ids` 调用追溯接口。

### 4.2 获取成分 → 口感映射（成分追溯）

```http
GET /api/teas/{tea_id}/component-flavor
```

用途：

返回某款茶的"成分 → 口感"映射列表，供前端"成分追溯"板块直观展示成分与口感的对应关系（如"美拉德反应产物 → 炒米香"）。每条论断挂证据溯源（`evidence` 指向论文）。

**注意**：这是第 1 层（知识 / 成分）→ 第 2 层（风味结构化）桥接关系的物化，**不是纵向追溯链的一层**。此处"追溯"指每条成分→口感论断的证据溯源，不进 `trace_links` 表、不新增 level；四层纵向追溯链仍由 `GET /api/trace/{output_id}` 提供。

响应示例：

```json
{
  "success": true,
  "data": {
    "tea_id": "BAMA_SZZ_TGY_NX",
    "links": [
      {
        "component": "美拉德反应产物",
        "component_category": "process_proxy",
        "flavor_key": "roasted",
        "flavor_label": "炒米香",
        "flavor_dimension": {"label_zh": "焙火感", "label_en": "Roasted Note", "intensity": 5},
        "mechanism": "浓香型焙火促进美拉德反应产物形成",
        "relationship": "primary_driver",
        "evidence": [
          {
            "id": "evidence_wang_2023_oolong",
            "source_type": "paper",
            "source": "Wang, D. et al. (2023). ...",
            "confidence": "medium",
            "note": "..."
          }
        ],
        "confidence": "medium",
        "notes": "焙火增加烘焙/焦糖样挥发物；非赛珍珠实际焙火温度"
      }
    ]
  },
  "meta": {"demo_mode": true, "fallback": false}
}
```

字段说明：

- `component`：成分名（挥发物 / 滋味成分 / 机理代理）。
- `component_category`：`aroma_compound`（挥发物）/ `taste_compound`（滋味成分）/ `process_proxy`（机理代理如美拉德、焙火路径）。
- `flavor_key`：指向该茶 `flavor-profile.dimensions[].key`（茶级归属、因茶而异）；若某成分对应不到现成 dimension 则为 `null`。
- `flavor_label`：UI 展示用的口感名（可能比 dimension 更细，如"兰花香" vs dimension `floral`）。
- `flavor_dimension`：从该茶 flavor_profile 按 `flavor_key` join 出的 `label_zh`/`label_en`/`intensity`；`flavor_key` 为空或不在该茶 dimensions 时为 `null`。
- `relationship`：`primary_driver`（主要造就）/ `contributes_to`（共同贡献）/ `participant`（参与）。
- `evidence`：该论断的证据明细列表（按 seed 声明顺序展开），每条含 `id`/`source_type`/`source`/`confidence`/`note`。
- `confidence`：`high` / `medium` / `low`。**如实区分直接支持(medium) vs 品类代理(low)**，低置信行 `notes` 注明"非单品实测值"。

前端建议：

- `flavor_dimension.intensity` 可与雷达图同维度对齐，用于把成分与雷达坐标关联展示。
- `confidence` 建议差异化呈现（如直接支持的论断与品类代理论断用不同视觉权重），避免把品类代理数据误读为单品实测。
- 需要进一步追溯证据来源时，`evidence[].id` 可与 `GET /api/teas/{tea_id}/knowledge` 返回的 evidence 对照。

## 5. 表达生成接口

说明：

表达生成接口内部会使用结构化规则库。前端不需要直接传入规则，也不需要拼接 prompt。后端会根据当前任务、茶品、市场、受众参照系和茶品术语，从 `generation_rules` 中筛选相关规则，再把少量匹配规则注入 LLM prompt。

例如：

```text
铁观音 × 欧美市场 × 精品咖啡爱好者 × 跨文化表达
```

后端会优先匹配：

```text
scope = cross_cultural_expression
market = western 或 any
audience_reference = specialty_coffee_lovers 或 any
trigger_terms 命中 观音韵 / 花香 / 回甘 等术语
```

这部分属于后端内部工作流，不作为前端必传字段。筛选出的规则 id 会通过响应 `meta.used_rule_ids` 暴露，便于前端展示"本输出受哪些规则约束"。

### 5.1 生成国内中文表达

```http
POST /api/teas/{tea_id}/domestic-expression
```

用途：

根据茶品和客户画像，生成面向国内消费者的中文表达。对应“感知 → 具象化语言”。

国内表达与跨文化表达（5.2）在表达生成层地位对等，是同一层的两条同构链路。跨文化表达由国内表达按规则横向翻译派生而来，规则库中会明确“国内的某个表达应如何信达雅地翻译成国外表达”。这一翻译关系为同层横向派生，不作为纵向追溯链的一层；通过跨文化表达响应中的 `source_expression_id` 字段记录，供前端可选展示。

国内表达属于 Demo 主路径，是跨文化表达的翻译源文，必须预置。

请求体示例：

```json
{
  "audience": {
    "age_group": "gen_z",
    "knowledge_level": "beginner",
    "scenario": "self_drinking",
    "psychology": "curiosity"
  },
  "style": "store_sales",
  "tone": "温润亲切",
  "length": "短（80字内）",
  "time_node": "中秋",
  "recipient": "送长辈"
}
```

`tone` / `length` / `time_node` / `task_type` / `flavor_reference` / `recipient` 为**可选 hint**（均默认 null，不传则不注入，行为同现状）：

| 字段 | 前端枚举 | 后端内部值（经 `app.enum_map` 翻译） |
|---|---|---|
| `tone`（语气） | 温润亲切 / 专业严谨 / 诗意古风 / 活泼年轻 / 商务克制 | `warm` / `professional` / `poetic` / `lively` / `restrained_business` |
| `length`（篇幅） | 短（80字内）/ 中（80-200字）/ 长（200字以上） | `short` / `medium` / `long` |
| `time_node`（时间节点） | 自由文本，**不经枚举映射、原样透传** | 原样（如"中秋""双11"） |
| `task_type`（任务类型） | `component-to-flavor` / `vague-to-vivid` | `component_to_flavor` / `vague_to_vivid`（连字符→下划线） |
| `flavor_reference`（风味参照体系） | `coffee` / `wine` / `none` | `coffee` / `wine` / `none`（自映射） |
| `recipient`（销售对象） | 自己喝 / 送长辈 / 送同事 / 送朋友 / 商务送礼 | `self` / `elder` / `colleague` / `friend` / `business_gifting` |

- hint 注入 prompt 影响话术调性 / 篇幅 / 场景化 / 生成方向 / 风味参照 / 销售对象场景，**不强制传**、不传不报错。
- 未知值（表外）不 422，原样透传 + warning；英文内部值自映射（前端传 `warm` 也通过）。
- hint 进 prompt → 进 `input_hash` 缓存键，**不同 hint 不命中同一缓存**。
- 响应回显**翻译后的内部值**（传"温润亲切"响应里是 `tone: "warm"`）；不传时响应不含该键。
- LLM 未启用时 hint 不影响 seed 兜底文案，但响应仍回显 hint（便于前端核对）。
- `task_type` / `flavor_reference` / `recipient` 的英文标识符对 LLM 不直观，prompt 里会再翻成一句描述性中文（"成分→风味…""送长辈 → 话术偏尊重、健康、传统意涵"等）喂给 LLM——`enum_map` 只负责值归一化，不负责文案。

响应示例：

```json
{
  "success": true,
  "data": {
    "expression_id": "expr_cn_szz_tgy_nx",
    "tea_id": "BAMA_SZZ_TGY_NX",
    "audience": {
      "age_group": "gen_z",
      "knowledge_level": "beginner",
      "scenario": "self_drinking",
      "psychology": "curiosity"
    },
    "outputs": {
      "story_style": "赛珍珠的铁观音值得用三种香气来理解：先闻干茶，是温暖的炒米香；再闻茶汤，是果甜香飘出来；入口后，兰花香才慢慢展开。三香层层递进，不是一次全给你。",
      "scientific_style": "浓香型铁观音的'三香'来自做青和焙火共同作用。同类铁观音研究显示，橙花叔醇、茉莉内酯等挥发物可能参与花香和果甜香的形成。当前成分解释基于品类研究，不是赛珍珠单品的检测值。",
```

### 5.2 生成跨文化表达

```http
POST /api/teas/{tea_id}/cross-cultural-expression
```

用途：

根据茶品、目标语言和受众参照系，生成面向海外受众的表达。Demo 阶段优先支持英语和欧美精品咖啡爱好者。

跨文化表达与国内表达（5.1）在表达生成层地位对等，属同一层的另一条同构链路。跨文化表达由国内表达按规则横向翻译派生而来：后端先读取国内表达作为翻译源文，再结合风味坐标和筛选出的规则，把中文 articulation 信达雅地转译为目标语言，并在跨文化链上额外进行跨文化类比适配（如用咖啡风味作参照、附加文化叙事），作为跨文化表达对象内部的子结构返回。响应中包含 `source_expression_id`，指向其翻译源文；该翻译关系为同层横向派生，不进入纵向追溯链，是否在 UI 上展示由前端决定。

请求体示例：

```json
{
  "target_language": "en",
  "market": "western",
  "audience_reference": "specialty_coffee_lovers",
  "audience_level": "beginner",
  "preserve_chinese_terms": true,
  "tone": "专业严谨",
  "length": "中（80-200字）",
  "time_node": "圣诞节",
  "recipient": "商务送礼"
}
```

`tone` / `length` / `time_node` / `task_type` / `flavor_reference` / `recipient` 与 5.1 国内表达**同一套枚举与映射规则**（见 5.1 表格），均为可选 hint，注入 prompt 影响跨文化话术的调性 / 篇幅 / 场景化 / 生成方向 / 风味参照 / 销售对象场景。`time_node` 原样透传（如"圣诞节""Prime会员日"）；`flavor_reference` 决定跨文化类比时参照哪个风味体系（coffee/wine/none）；`recipient`（如"送长辈""商务送礼"）由 LLM 结合 `{market}` 市场语境做文化适配。

响应示例：


字段说明：

- `source_expression_id`：本次跨文化表达横向翻译派生自的国内表达 ID，为同层横向派生引用，不作为纵向追溯链的一层。是否在 UI 上展示由前端决定。

### 5.3 自然语言生成表达（NL 入口）

> **【搁置】** 本入口已实现并通过测试，当前前端不接入：前端输入框为受控词输入而非自由文本，与 NL 入口假设的自由文本形态不匹配。端点仍注册、仍可调、测试仍跑，代码全部保留。决策记录见 `docs/compromises.md`「NL 自然语言入口」。后续前端若引入自由文本输入框可直接复活。

```http
POST /api/natural-expression
```

用途：

让用户用一段自由文本描述想要的表达，后端自动识别茶品并复用现有表达链路生成话术，免去手动传 `tea_id` 和选链路。典型输入如"给第一次喝铁观音的顾客写一段亲切、简短的三香介绍""用英文给西方咖啡爱好者介绍铁观音"。

后端处理流程：

1. 调一次意图解析 LLM，把自然语言解析成 `{tea_id, chain}`：
   - `tea_id` 从 DB 茶品枚举（当前铁观音 / 大红袍 / 金骏眉）中识别；未提及或提了清单外的茶 → `null`。新增茶品自动跟进枚举，无需维护。
   - `chain` 默认 `domestic`；仅当用户明确要求英文 / 西方 / 海外受众 / translate to English / for Westerners / overseas 等跨文化诉求时取 `cross_cultural`。
   - 受众 / 风格 / 语气 / 篇幅 / 侧重等细节不在此抽取，留在原始 NL 里作 directive 透传给话术 LLM。
2. 按 `chain` 路由到对应链路的现有生成逻辑，原始 NL 作为 `directive` 注入 prompt：
   - `domestic` → 复用 5.1 国内表达（`story_style` / `scientific_style` / `emotional_style`）。
   - `cross_cultural` → 复用 5.2 跨文化表达（`literal_explanation` / `beginner_analogy` / `cultural_narrative` + `analogy_rules`），用唯一支持的 en / western / specialty_coffee_lovers 组合，`source_expression_id` 仍指向国内 seed 记录。
3. 响应 shape 与对应链路完全一致，前端按 `meta.nl.chain` 选渲染器（两个 shape 前端都已有渲染器）。

请求体示例：

```json
{
  "text": "给第一次喝铁观音的顾客写一段亲切、简短的三香介绍"
}
```

响应示例（chain=domestic，shape 同 5.1）：

```json
{
  "success": true,
  "data": {
    "expression_id": "expr_cn_szz_tgy_nx",
    "tea_id": "BAMA_SZZ_TGY_NX",
    "audience": {},
    "outputs": {
      "story_style": "...",
      "scientific_style": "...",
      "emotional_style": "..."
    },
    "source_profile_id": "flavor_szz_tgy_nx",
    "trace_id": "expr_cn_szz_tgy_nx"
  },
  "meta": {
    "demo_mode": true,
    "fallback": false,
    "llm_generated": true,
    "used_rule_ids": ["rule_domestic_store_sales"],
    "nl": {
      "directive": "给第一次喝铁观音的顾客写一段亲切、简短的三香介绍",
      "intent_llm_generated": true,
      "chain": "domestic"
    }
  }
}
```

响应示例（chain=cross_cultural，shape 同 5.2）：

```json
{
  "success": true,
  "data": {
    "translation_id": "expr_en_szz_tgy_nx_coffee",
    "tea_id": "BAMA_SZZ_TGY_NX",
    "target_language": "en",
    "market": "western",
    "audience_reference": "specialty_coffee_lovers",
    "outputs": {
      "literal_explanation": "...",
      "beginner_analogy": "...",
      "cultural_narrative": "..."
    },
    "analogy_rules": [],
    "source_profile_id": "flavor_szz_tgy_nx",
    "source_expression_id": "expr_cn_szz_tgy_nx",
    "trace_id": "expr_en_szz_tgy_nx_coffee"
  },
  "meta": {

    "fallback": false,
    "llm_generated": true,

    }
  }
}
```

`meta.nl` 字段说明：

- `directive`：回显用户原始输入（供前端展示用户说了什么）。
- `intent_llm_generated`：意图解析是否走了 LLM（含缓存命中）。意图结果按 input_hash 缓存进 `generated_outputs` 表，同输入二次调用命中即跳过 LLM 调用。
- `chain`：本次路由到的链路，前端据此选择对应链路的渲染器。

fallback 场景：

- 未配置 LLM（`LLM_API_KEY` / `LLM_BASE_URL` 缺失）：NL 意图解析无法 seed 兜底，整个接口返回 `feature_not_available` fallback。这与现有三接口（5.1 / 5.2 / 6.1）未启用时走 seed 兜底是不同情况——NL 链路上 seed 兜底无意义。
- 意图解析 LLM 调用失败（网络 / 超时 / 解析错误 / 网关错误）：返回 fallback。
- 识别到的茶不在 Demo 支持范围（如"给龙井写个介绍"）：`tea_id=null`，返回 fallback，提示当前支持的茶品。
- 话术生成阶段返回未预置 / 不支持组合：返回 fallback。

已知边角：

- 用户要求跨文化但指定非支持组合（如"翻译成日语"），意图仍判 `cross_cultural`，后端用唯一支持的英文组合生成，directive 里的"日语"进 prompt 但输出语言为英文。Demo 阶段可接受，不为此扩 combo。
- NL 生成的表达复用现有 seed 记录，`trace_id` / `source_profile_id` / `source_expression_id` 照旧来自 seed，纵向追溯链不断。NL 层只是表达接口的另一个入口，不进追溯链、不新增 trace 节点。

## 6. 营销物料接口

### 6.1 生成图片物料

```http
POST /api/teas/{tea_id}/marketing-asset
```

用途：

生成营销图片物料所需的数据。物料接口本身不调生图（`meta.image_generation_enabled=false`），只产出海报文案、雷达图数据、图片生成 prompt，真实出图走独立接口 §6.2。

国内链与跨文化链各自生成物料，方式相同，区别仅在 `language` 和文化。`language=zh` 时读取国内表达生成中文物料，`language=en` 时读取跨文化表达生成英文物料。

请求体示例（跨文化 / 英文）：

```json
{
  "route_id": "szz_western_coffee_poster",
  "asset_type": "poster",
  "platform": "tiktok",
  "language": "en",
  "style": "premium_but_approachable"
}
```

请求体示例（国内 / 中文，传中文枚举）：

```json
{
  "route_id": "szz_domestic_poster",
  "asset_type": "poster",
  "platform": "小红书",
  "language": "zh",
  "style": "国风",
  "content_theme": "tea-culture"
}
```

`platform` / `style` / `content_theme` 字段：前端可传**中文枚举**（`content_theme` 除外，前端 value 已是英文短横线），后端按 `app.enum_map` 翻成**内部英文值**（映射集中在 `backend/app/enum_map.py` 一处，前端加枚举时后端改一处）：

| 字段 | 前端枚举 | 后端内部英文值 |
|---|---|---|
| `platform`（国内） | 小红书 / 抖音 / 微信视频号 | `xiaohongshu` / `douyin` / `wechat_channels` |
| `platform`（海外） | Instagram / TikTok / YouTube | `instagram` / `tiktok` / `youtube` |
| `style`（物料风格） | 年轻 / 商务 / 国风 | `youthful` / `business` / `guofeng` |
| `content_theme`（内容主题） | `tea-marketing` / `tea-culture`（前端 value） | `tea_marketing` / `tea_culture`（连字符→下划线） |

- **抖音 ≠ TikTok**：同一 app，但国内投放偏下沉、海外偏年轻，让 LLM 区别对待，不合并成同一个内部值。
- **未知值原样透传**：传了表外的值（如"微博"）不 422、不丢弃，记一条 warning 后原样透传给下游（Demo 友好，前端临时新增枚举不阻断）。
- **英文内部值自映射**：前端直接传 `tiktok` / `guofeng` / `tea_culture` 等内部值也原样通过，不误判为未知。
- **`content_theme`**：决定文案侧重——`tea_marketing` 偏卖点/促销/礼盒场景，`tea_culture` 偏茶道/产地/工艺叙事。不传时 LLM 默认偏营销。注入 prompt，进缓存键。
- **响应回显**：响应里 `data.platform` / `data.style` / `data.content_theme` 回显的是**翻译后的内部英文值**，不回显中文枚举；不传时不含该键。
- **`style` 两套维度别混**：本接口的 `style`（年轻/商务/国风）管**文案调性**，与生图接口 §6.2 的 `style`（`fresh`/`business`，管画面光照色调）是两套独立维度，不可混用。

### 6.2 生成图片（生图）

```http
POST /api/image/generate
```

用途：

调用豆包 Seedream（火山方舟 Ark）文生图，返回真实图片 URL，并**把知识点 + 产品文案（中文 copy）直接渲染在图上**。与 §6.1 营销物料**两步联调**：先调 `marketing-asset` 拿 `image_prompt`，再把该 prompt 传给本接口出图。物料层只产 prompt，不直接出图——解耦海报文案与图片生成的耗时（Seedream pro 2K 出图偏慢，首图常 >90s，且超时也计费，`IMAGE_TIMEOUT` 默认 300s）。

> **图源已切换为豆包 Seedream（2026-07-15）**：原 CogView-4 中文文字渲染能力不稳，而海报初衷是把"知识点 + 产品文案"印在图上，纯无字照片偏题。Seedream 图内中文渲染准确，`response_format="url"` 直接返临时 URL（无需改 b64 落盘架构）。图内中文 copy 由后端按 `tea_id + language` 从 seed asset 表取（`headline/subheadline/body`）；传 `tea_id + language` 即印字，不传则纯画面出图。`scene`（closeup / landscape / product）× `style`（fresh / business）维度保留，缓存键含 `(prompt, size, style, scene, language)` 防投毒。

请求体示例：

```json
{
  "prompt": "Saizhenzhu Tieguanyin oolong tea. A white porcelain gaiwan filled with clear golden tea liquor, with rolled Tieguanyin tea leaves and a few pale orchid flowers arranged beside it. The tea originates from Anxi, Fujian.",
  "size": "2K",
  "style": "fresh",
  "scene": "closeup",
  "tea_id": "BAMA_SZZ_TGY_NX",
  "language": "zh",
  "route_id": "szz_western_coffee_poster"
}
```

字段说明：

- `prompt`（必填）：图片生成 prompt，通常取自 `marketing-asset.image_prompt`。建议只写画面物体 + 产地线索，镜头/光照/色调/氛围交给 `scene` 与 `style`（不要写 `no generated text`——图内文字由后端单独渲染）。
- `size`（可选）：输出尺寸，Ark 用档位字符串如 `2K` / `1K`；空用配置默认 `IMAGE_SIZE`。
- `style`（可选，默认 `fresh`）：海报风格。`fresh` = 清新晨光 / 浅绿象牙色 / 通透自然背景；`business` = 暗调影棚光 / 深炭金 / 深色奢华背景（要商务时显式传）。
- `scene`（可选，默认 `closeup`）：镜头。`closeup` = 茶具茶汤特写；`landscape` = 产地广角山林（人物/茶具在下、山林在上）；`product` = 商品罐图主体。`scene` 与 `style` 正交，可任意组合。
- `tea_id`（可选）：溯源回显；兼配合 `language` 取 seed copy 印进图。
- `route_id`（可选）：仅作溯源回显，不参与生图。
- `language`（可选，默认 `null`）：`zh` / `en`。配合 `tea_id` 时后端从 seed asset 表取 copy（`headline/subheadline/body`）渲染进图；不传或取不到 copy 则纯画面出图（图里没字，不抛、不 fallback）。

响应示例（成功）：

```json
{
  "success": true,
  "data": {
    "image_url": "https://ark-files.volces.com/.../xxxx.png",
    "prompt": "Saizhenzhu Tieguanyin oolong tea. A white porcelain gaiwan ...",
    "model": "doubao-seedream-5-0-pro-260628",
    "size": "2K",
    "style": "fresh",
    "scene": "closeup",
    "language": "zh",
    "tea_id": "BAMA_SZZ_TGY_NX",
    "route_id": "szz_western_coffee_poster"
  },
  "meta": {
    "demo_mode": true,
    "fallback": false,
    "image_generated": true
  }
}
```

响应示例（未启用 / 失败 → fallback）：

```json
{
  "success": true,
  "data": {
    "title": "生图未启用",
    "message": "未配置 IMAGE_API_KEY / IMAGE_BASE_URL，生图不可用。请在 backend/.env 填豆包 Seedream（火山方舟 Ark）凭证后重启。",
    "available_route_id": "szz_western_coffee_poster"
  },
  "meta": {
    "demo_mode": true,
    "fallback": true,
    "fallback_reason": "image_not_enabled"
  }
}
```

字段说明：

- `image_url`：Ark 返回的图片临时链接，**有效期 30 天**，前端如需长期展示请自行下载转存。同一 prompt+size+style+scene+language 的结果按 input_hash 缓存（29 天内复用、跳过 Seedream 调用；style / scene / language 都是缓存键的一部分，切换各自独立缓存）。
- `model` / `size` / `style` / `scene` / `language`：实际使用的模型、尺寸、风格、镜头、copy 语言（缓存命中时沿用缓存值）。`language` 为 `null` 表示纯画面出图（未取 copy）。
- `tea_id` / `route_id`：仅当请求传入时回显。
- `meta.image_generated=true`：本次返回了真实图片。未启用 / 失败时 `fallback=true` 且 `fallback_reason ∈ image_not_enabled / timeout / network_error / gateway_error / parse_error`，不返回 `image_url`。
- 缺 `prompt` → 422 `VALIDATION_ERROR`（走统一错误响应）。

**prompt 富化（图内渲染文字）**：后端在发图前给 `prompt` 套确定性片段 + 图内中文文字，不调 LLM、零幻觉：
1. `prompt`（来自 `marketing-asset.image_prompt`）：画面物体 + 产地线索（茶具 / 茶汤 / 道具 / 产地）。
2. `scene` 片段：镜头与构图（`closeup` 特写 / `landscape` 产地广角 / `product` 商品罐图）+ 竖版 9:16。
3. `style` 片段：光照 + 色调 + 氛围 + 背景——`fresh`（清新晨光 / 浅绿象牙 / 通透背景）或 `business`（暗调影棚光 / 深炭金 / 深色奢华背景）。
4. 图内中文文字段（传 `tea_id + language` 时注入）：`headline` / `subheadline` / `body` 直接叠在照片上、full-bleed 无白边、带暗色渐变保证可读。
5. 画质后缀：浅景深 / 高细节 / 8k / photorealistic。

刻意**不含** `Professional commercial product photography / elegant composition` 这类企业画册美学词（实测会把出图拽向"商务老气"风）；要商务感由调用方显式传 `style=business` 触发商务片段。`scene` 与 `style` 正交，同一茶 prompt 可组合出 3 × 2 = 6 种出图方向，缓存键含 `(prompt, size, style, scene, language)` 防投毒。同时关闭水印（`extra_body.watermark=false`）。Seedream 无 `quality` 参数（CogView 的 `quality=hd` 已废弃）。

注意：图内中文知识文字由后端从 seed asset 表取 copy 直接渲染进图（Seedream 中文渲染稳定）；`marketing-asset` 返回的 `copy` 字段同样含 `headline/subheadline/body`，前端可直接用作文案展示。LLM 启用且改了文案时，图内印的 seed copy 与前端显示的 LLM copy 可能不一致——demo 阶段接受。

配置（`backend/.env`，与 `LLM_*` 相互独立）：

```
IMAGE_API_KEY=<ark- 开头的 ARK API Key>
IMAGE_BASE_URL=https://ark.cn-beijing.volces.com/api/v3
IMAGE_MODEL=doubao-seedream-5-0-pro-260628
IMAGE_SIZE=2K
IMAGE_QUALITY=
IMAGE_TIMEOUT=300
```

prompt 富化在后端内部完成（确定性片段 + 图内文字），前端无需关心。需在火山方舟控制台开通 `doubao-seedream-5-0-pro` 模型并关闭"安全体验模式"推理限额，否则 429 `SetLimitExceeded`。视频生成仍为 P2 占位（`POST /api/teas/{tea_id}/video-asset` 返回 fallback），不在本期开放。

## 9. 调试接口

调试接口用于 Demo 当场确认 LLM 是否接上、是否可用，**非 P0 契约接口**，前端不应依赖其存在或字段稳定。

### 9.1 LLM 健康检查

```http
GET /api/health-llm
```

用途：

确认后端是否已正确加载 LLM 配置（API key / base_url / 模型 / JSON mode）。LLM 未启用时，三个文本生成接口（5.1 / 5.2 / 6.1）会自动走 seed 兜底，行为与未接 LLM 时一致；该接口用于在 Demo 现场快速区分「LLM 未接」与「LLM 接了但调用失败」。注意：生图（§6.2）走独立的 `IMAGE_*` 配置，不经此接口反映。

响应示例（已配置 key / base_url）：

```json
{
  "success": true,
  "data": {
    "llm_enabled": true,
    "llm_model": "glm-5.2",
    "llm_base_url_masked": "https://open.bigmodel.cn/**",
    "llm_supports_json_mode": true
  },
  "meta": {
    "demo_mode": true,
    "fallback": false
  }
}
```

响应示例（未配置 key / base_url）：

```json
{
  "success": true,
  "data": {
    "llm_enabled": false,
    "llm_model": "glm-5.2",
    "llm_base_url_masked": "",
    "llm_supports_json_mode": true
  },
  "meta": {
    "demo_mode": true,
    "fallback": false
  }
}
```

字段说明：

- `llm_enabled`：是否已同时配置 `LLM_API_KEY` 与 `LLM_BASE_URL`。`false` 时三个文本生成接口（5.1 / 5.2 / 6.1）自动走 seed 兜底，响应 `meta.llm_generated=false` 且**不输出** `llm_fallback_reason`（与未接 LLM 时一致）。注意：生图（§6.2）走独立的 `IMAGE_*` 配置，其启用状态不经此字段反映。
- `llm_model`：当前使用的模型名（默认 `glm-5.2`）。
- `llm_base_url_masked`：仅暴露 `scheme://host`，掩码 path / query，**不泄露完整端点**。
- `llm_supports_json_mode`：代理是否支持 OpenAI `response_format={"type":"json_object"}`。`false` 时后端走防御式 JSON 解析兜底。

注意：该接口**不输出明文 API key**，key 仅存在于本地 `.env`（gitignored），不进任何被跟踪文件或镜像。

## 11. 实现原则

1. SQLite 数据库是运行时读真源与 LLM 输出缓存，不作为人工维护的数据源；由 `seed.py --reset` 从 YAML 灌表。
2. Demo 数据应来自可提交到 Git 的 YAML 或 JSON seed 文件。
3. 关键资料应带 `source_type`、`source`、`confidence` 等轻量证据字段；Demo 阶段这些字段由团队人工确认，Agent 只辅助整理。
4. 生成规则应作为结构化数据维护在 seed 文件和 SQLite 中，例如 `generation_rules`；不要长期硬编码在 Python 或超长 prompt 中。
5. 后端生成表达时，应按任务、市场、受众和触发术语筛选少量相关规则，再注入 prompt。
6. 所有 LLM 生成结果必须尽量包含 `source_profile_id` 或 `trace_id`；跨文化表达必须额外包含 `source_expression_id`，记录其横向翻译派生自的国内表达，该关系不进入纵向追溯链。
7. Demo 阶段不要让未开放功能报错或白屏，统一返回 fallback。
8. 前端不要依赖自然语言解析，所有关键内容都应放在稳定 JSON 字段中。NL 入口（§5.3）只是表达接口的另一个入口，最终响应 shape 与对应链路一致、关键字段仍来自 seed，不依赖 NL 解析结果。
9. 后端可以先返回静态 mock 数据，再逐步替换为数据库查询、规则筛选和 LLM 调用。（读路径已切库：getter 查 SQLite；写路径经 output_store 缓存 LLM 输出。）
