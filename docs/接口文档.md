# Demo 接口文档

本文档用于前后端协作，描述 Demo 阶段后端 API 的功能边界、请求参数、响应结构和 fallback 规则。

Demo 当前主路径为：

```text
1 款茶（铁观音）× 图片物料 ×（国内链 + 跨文化链）两条同构链路
```

两条链路均完整跑通：国内链面向国内消费者，跨文化链面向欧美精品咖啡爱好者。两条链共享同一款茶的知识与风味坐标，跨文化表达由国内表达按规则横向翻译派生而来，这一关系通过 `source_expression_id` 字段记录，不进入纵向追溯链。

后端会优先保证这两条链路完整跑通。其他茶品、市场、受众参照系、视频生成等能力可以先由前端做好入口，后端通过 fallback 返回“暂未开放”的标准响应，避免页面空白或报错。

## 1. 通用约定

### 1.1 Base URL

本地开发默认：

```text
http://localhost:8000
```

所有业务接口统一以 `/api` 开头。

### 1.2 数据格式

请求和响应均使用 JSON：

```http
Content-Type: application/json
```

### 1.3 ID 约定

Demo 阶段建议使用可读字符串 ID，而不是纯数字 ID，方便前后端调试和追溯。

示例：

```text
BAMA_SZZ_TGY_NX
flavor_szz_tgy_nx
expr_cn_szz_tgy_nx
expr_en_szz_tgy_nx_coffee
asset_szz_poster_zh
asset_szz_poster_en
```

### 1.4 通用响应字段

正常响应建议包含：

```json
{
  "success": true,
  "data": {},
  "meta": {
    "demo_mode": true,
    "fallback": false,
    "llm_generated": true,
    "used_rule_ids": ["rule_domestic_store_sales_style"]
  }
}
```

生成类接口（5.1 / 5.2 / 6.1）的 `meta` 额外字段：

- `llm_generated`：本次输出是否由 LLM 真实生成。`false` 表示退回了 seed 预置表达（未启用或调用失败时的兜底）。
- `llm_fallback_reason`：仅当 **LLM 已启用（`llm_enabled=true`）但本次未产出 LLM 内容**（`llm_generated=false`）时出现。未启用 LLM 时 `llm_generated=false` 但**不输出**本字段（行为与未接 LLM 时一致）。取值分两类：**(1) 尝试过 LLM 但失败** —— `network_error`（连接层失败：DNS / 握手 / 断流）/ `timeout`（请求超时）/ `parse_error`（LLM 输出非 JSON 或校验失败）/ `gateway_error`（网关返回 4xx/5xx：内容审查 / 额度 / 模型不存在 / 上下文超限等）；**(2) 调 LLM 前源文缺失，未真正调用** —— `domestic_source_missing`（跨文化链取国内表达作翻译源文，但国内 seed 不存在）/ `expression_source_missing`（物料层取对应语言表达作文案依据，但该表达 seed 不存在）。两类均主路径不可达、Demo 阶段不会触发，仅作防御性兜底。注意：LLM→seed 降级**不**置 `fallback=true`，仍属成功响应，前端可直接使用 seed 文本展示。
- `used_rule_ids`：本次注入 LLM prompt 的结构化规则 id 列表，用于"规则约束生成"可追溯。

fallback 响应建议包含：

```json
{
  "success": true,
  "data": {},
  "meta": {
    "demo_mode": true,
    "fallback": true,
    "fallback_reason": "该功能 Demo 阶段暂未开放"
  }
}
```

错误响应建议包含：

```json
{
  "success": false,
  "error": {
    "code": "TEA_NOT_FOUND",
    "message": "未找到对应茶品"
  }
}
```

## 5. 表达生成接口

说明：

表达生成接口内部会使用结构化规则库。前端不需要直接传入规则，也不需要拼接 prompt。后端会根据当前任务、茶品、市场、受众参照系和茶品术语，从 `generation_rules` 中筛选相关规则，再把少量匹配规则注入 LLM prompt。

例如：

```text
铁观音 × 欧美市场 × 精品咖啡爱好者 × 跨文化表达
```

后端会优先匹配：

```text
scope = cross_cultural_expression
market = western 或 any
audience_reference = specialty_coffee_lovers 或 any
trigger_terms 命中 观音韵 / 花香 / 回甘 等术语
```

这部分属于后端内部工作流，不作为前端必传字段。筛选出的规则 id 会通过响应 `meta.used_rule_ids` 暴露，便于前端展示"本输出受哪些规则约束"。

### 5.1 生成国内中文表达

```http
POST /api/teas/{tea_id}/domestic-expression
```

用途：

根据茶品和客户画像，生成面向国内消费者的中文表达。对应“感知 → 具象化语言”。

国内表达与跨文化表达（5.2）在表达生成层地位对等，是同一层的两条同构链路。跨文化表达由国内表达按规则横向翻译派生而来，规则库中会明确“国内的某个表达应如何信达雅地翻译成国外表达”。这一翻译关系为同层横向派生，不作为纵向追溯链的一层；通过跨文化表达响应中的 `source_expression_id` 字段记录，供前端可选展示。

国内表达属于 Demo 主路径，是跨文化表达的翻译源文，必须预置。

请求体示例：

```json
{
  "audience": {
    "age_group": "gen_z",
    "knowledge_level": "beginner",
    "scenario": "self_drinking",
    "psychology": "curiosity"
  },
  "style": "store_sales"
}
```

响应示例：

```json
{
  "success": true,
  "data": {
    "expression_id": "expr_cn_szz_tgy_nx",
    "tea_id": "BAMA_SZZ_TGY_NX",
    "audience": {
      "age_group": "gen_z",
      "knowledge_level": "beginner",
      "scenario": "self_drinking",
      "psychology": "curiosity"
    },
    "outputs": {
      "story_style": "赛珍珠的铁观音值得用三种香气来理解：先闻干茶，是温暖的炒米香；再闻茶汤，是果甜香飘出来；入口后，兰花香才慢慢展开。三香层层递进，不是一次全给你。",
      "scientific_style": "浓香型铁观音的'三香'来自做青和焙火共同作用。同类铁观音研究显示，橙花叔醇、茉莉内酯等挥发物可能参与花香和果甜香的形成。当前成分解释基于品类研究，不是赛珍珠单品的检测值。",
      "emotional_style": "下午泡一杯赛珍珠，第一口是炒米的温暖感，慢慢花香透出来，喝完喉咙里甜感又重新浮现。这就是铁观音迷人的地方——它不会一次把答案全告诉你。"
    },
    "source_profile_id": "flavor_szz_tgy_nx",
    "trace_id": "expr_cn_szz_tgy_nx"
  },
  "meta": {
    "demo_mode": true,
    "fallback": false
  }
}
```

### 5.2 生成跨文化表达

```http
POST /api/teas/{tea_id}/cross-cultural-expression
```

用途：

根据茶品、目标语言和受众参照系，生成面向海外受众的表达。Demo 阶段优先支持英语和欧美精品咖啡爱好者。

跨文化表达与国内表达（5.1）在表达生成层地位对等，属同一层的另一条同构链路。跨文化表达由国内表达按规则横向翻译派生而来：后端先读取国内表达作为翻译源文，再结合风味坐标和筛选出的规则，把中文 articulation 信达雅地转译为目标语言，并在跨文化链上额外进行跨文化类比适配（如用咖啡风味作参照、附加文化叙事），作为跨文化表达对象内部的子结构返回。响应中包含 `source_expression_id`，指向其翻译源文；该翻译关系为同层横向派生，不进入纵向追溯链，是否在 UI 上展示由前端决定。

请求体示例：

```json
{
  "target_language": "en",
  "market": "western",
  "audience_reference": "specialty_coffee_lovers",
  "audience_level": "beginner",
  "preserve_chinese_terms": true
}
```

响应示例：

```json
{
  "success": true,
  "data": {
    "translation_id": "expr_en_szz_tgy_nx_coffee",
    "tea_id": "BAMA_SZZ_TGY_NX",
    "target_language": "en",
    "market": "western",
    "audience_reference": "specialty_coffee_lovers",
    "outputs": {
      "literal_explanation": "Saizhenzhu is Bama Tea's flagship Tieguanyin — a roasted oolong with toasted-rice warmth, a fruity-sweet lift in the aroma, and an orchid-like floral note that unfolds on the palate.",
      "beginner_analogy": "Think of the contrast in a carefully roasted coffee: warm toasted notes first, then a cleaner floral finish — but with the lighter body and long hui gan (returning sweetness) of oolong tea.",
      "cultural_narrative": "Its defining character is called Guanyin Yun. It is not a single aroma, but the lingering unity of the Tieguanyin cultivar, Anxi terroir, traditional craft and aftertaste. The three-aroma progression — toasted rice, fruity sweetness, orchid — is Bama's way of guiding drinkers into this depth."
    },
    "analogy_rules": [
      {
        "source_dimension": "花香",
        "target_reference": "floral finish in washed coffee",
        "confidence": "medium",
        "note": "用于降低理解门槛，不等同于完全相同的风味物质"
      }
    ],
    "source_profile_id": "flavor_szz_tgy_nx",
    "source_expression_id": "expr_cn_szz_tgy_nx",
    "trace_id": "expr_en_szz_tgy_nx_coffee"
  },
  "meta": {
    "demo_mode": true,
    "fallback": false
  }
}
```

字段说明：

- `source_expression_id`：本次跨文化表达横向翻译派生自的国内表达 ID，为同层横向派生引用，不作为纵向追溯链的一层。是否在 UI 上展示由前端决定。
- `source_profile_id`：风味坐标 ID，跨文化表达的结构化风味依据，纵向追溯链的上一级。
- `trace_id`：本输出自身的追溯 ID，可用于 `GET /api/trace/{output_id}`。

fallback 场景：

- `target_language` 不是 `en`。
- `market` 不是 `western`。
- `audience_reference` 不是 `specialty_coffee_lovers`。

这些场景下后端可以返回标准 fallback，而不是报错。

## 6. 营销物料接口

### 6.1 生成图片物料

```http
POST /api/teas/{tea_id}/marketing-asset
```

用途：

生成营销图片物料所需的数据。Demo 阶段不强依赖真实生图 API，后端可以返回海报文案、雷达图数据、图片生成 prompt，由前端模板渲染。

国内链与跨文化链各自生成物料，方式相同，区别仅在 `language` 和文化。`language=zh` 时读取国内表达生成中文物料，`language=en` 时读取跨文化表达生成英文物料。

## 8. Fallback 接口与规则

### 8.1 通用 fallback

```http
GET /api/fallback
POST /api/fallback
```

用途：

当前端访问尚未开放的功能时，返回统一占位内容。适用于视频生成、多语言扩展、非 Demo 茶品、非咖啡受众参照系等。

请求体示例：

```json
{
  "feature": "video_generation",
  "requested_path": "/api/teas/BAMA_SZZ_TGY_NX/video-asset",
  "reason": "frontend_placeholder"
}
```

响应示例：

```json
{
  "success": true,
  "data": {
    "title": "功能暂未开放",
    "message": "该能力已在产品规划中，Demo 阶段暂不提供真实生成结果。",
    "suggested_action": "请返回当前 Demo 主路径：铁观音 × 图片物料 ×（国内链 + 跨文化链）。",
    "available_route_id": "szz_western_coffee_poster"
  },
  "meta": {
    "demo_mode": true,
    "fallback": true,
    "fallback_reason": "feature_not_available"
  }
}
```

### 8.2 未开放功能的推荐路由

以下接口可以先不实现真实逻辑，但建议后端注册路由并返回 fallback，避免前端点击时报 404。

```http
POST /api/teas/{tea_id}/video-asset
POST /api/translate
POST /api/image/generate
POST /api/audio/generate
```

推荐响应：

```json
{
  "success": true,
  "data": {
    "title": "功能暂未开放",
    "message": "当前 Demo 暂不支持该功能，后续可由后端接入真实能力。",
    "available_route_id": "szz_western_coffee_poster"
  },
  "meta": {
    "demo_mode": true,
    "fallback": true,
    "fallback_reason": "feature_not_available"
  }
}
```

### 8.3 市场与受众参照系枚举（已实现）

`GET /api/markets` 与 `GET /api/audience-references` 已从 P2 占位 fallback 升级为真实列表，供前端选择器使用。两者均从 `demo_routes.yaml` 派生去重，附双语展示名（label 来源为 Demo 阶段展示名映射，待真市场/受众 catalog 落地后迁入 seed）。

{

字段说明：

### 8.4 404 自动 fallback

后端可以设置全局 404 处理器。当前端访问不存在的 `/api/*` 路由时，不直接返回默认 404，而是返回 fallback JSON。

建议响应：

```json
{
  "success": true,
  "data": {
    "title": "接口暂未开放",
    "message": "该接口尚未在 Demo 后端中实现。请确认是否属于后续功能。",
    "available_route_id": "szz_western_coffee_poster"
  },
  "meta": {
    "demo_mode": true,
    "fallback": true,
    "fallback_reason": "api_not_implemented"
  }
}
```

注意：

只建议对 `/api/*` 做 fallback。非 API 路由仍可按前端框架自己的路由规则处理。

## 9. 调试接口

调试接口用于 Demo 当场确认 LLM 是否接上、是否可用，**非 P0 契约接口**，前端不应依赖其存在或字段稳定。

### 9.1 LLM 健康检查

```http
GET /api/health-llm
```

用途：

确认后端是否已正确加载 LLM 配置（API key / base_url / 模型 / JSON mode）。LLM 未启用时，三个生成接口会自动走 seed 兜底，行为与未接 LLM 时一致；该接口用于在 Demo 现场快速区分「LLM 未接」与「LLM 接了但调用失败」。

响应示例（已配置 key / base_url）：

```json
{
  "success": true,
  "data": {
    "llm_enabled": true,
    "llm_model": "glm-5.2",
    "llm_base_url_masked": "https://open.bigmodel.cn/**",
    "llm_supports_json_mode": true
  },
  "meta": {
    "demo_mode": true,
    "fallback": false
  }
}
```

响应示例（未配置 key / base_url）：

```json
{
  "success": true,
  "data": {
    "llm_enabled": false,
    "llm_model": "glm-5.2",
    "llm_base_url_masked": "",
    "llm_supports_json_mode": true
  },
  "meta": {
    "demo_mode": true,
    "fallback": false
  }
}
```

字段说明：

- `llm_enabled`：是否已同时配置 `LLM_API_KEY` 与 `LLM_BASE_URL`。`false` 时三个生成接口自动走 seed 兜底，响应 `meta.llm_generated=false` 且**不输出** `llm_fallback_reason`（与未接 LLM 时一致）。
- `llm_model`：当前使用的模型名（默认 `glm-5.2`）。
- `llm_base_url_masked`：仅暴露 `scheme://host`，掩码 path / query，**不泄露完整端点**。
- `llm_supports_json_mode`：代理是否支持 OpenAI `response_format={"type":"json_object"}`。`false` 时后端走防御式 JSON 解析兜底。

注意：该接口**不输出明文 API key**，key 仅存在于本地 `.env`（gitignored），不进任何被跟踪文件或镜像。

## 11. 实现原则

1. SQLite 数据库只作为运行产物，不作为人工维护的数据源。
2. Demo 数据应来自可提交到 Git 的 YAML 或 JSON seed 文件。
3. 关键资料应带 `source_type`、`source`、`confidence` 等轻量证据字段；Demo 阶段这些字段由团队人工确认，Agent 只辅助整理。
4. 生成规则应作为结构化数据维护在 seed 文件和 SQLite 中，例如 `generation_rules`；不要长期硬编码在 Python 或超长 prompt 中。
5. 后端生成表达时，应按任务、市场、受众和触发术语筛选少量相关规则，再注入 prompt。
6. 所有 LLM 生成结果必须尽量包含 `source_profile_id` 或 `trace_id`；跨文化表达必须额外包含 `source_expression_id`，记录其横向翻译派生自的国内表达，该关系不进入纵向追溯链。
7. Demo 阶段不要让未开放功能报错或白屏，统一返回 fallback。
8. 前端不要依赖自然语言解析，所有关键内容都应放在稳定 JSON 字段中。
9. 后端可以先返回静态 mock 数据，再逐步替换为数据库查询、规则筛选和 LLM 调用。
